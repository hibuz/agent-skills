# 코드 리뷰 및 품질 (Code Review and Quality)

## 개요 (Overview)

품질 게이트(Quality gates)를 통한 다차원 코드 리뷰입니다. 모든 변경 사항은 머지 전에 리뷰를 거쳐야 합니다. 예외는 없습니다. 리뷰는 정확성(correctness), 가독성(readability), 아키텍처(architecture), 보안(security), 성능(performance)의 다섯 가지 축을 다룹니다.

**승인 기준:** 완벽하지 않더라도 전체적인 코드 건강성을 분명히 개선한다면 변경 사항을 승인하세요. 완벽한 코드는 존재하지 않습니다. 목표는 지속적인 개선입니다. 단지 당신이 작성했을 방식과 정확히 일치하지 않는다는 이유로 변경을 막지 마세요. 코드베이스를 개선하고 프로젝트 관례를 따른다면 승인하세요.

## 사용 시기 (When to Use)

- PR이나 변경 사항을 머지하기 전
- 기능 구현을 완료한 후
- 다른 에이전트나 모델이 생성한 코드를 평가해야 할 때
- 기존 코드를 리팩토링할 때
- 버그 수정 후 (수정 사항과 회귀 테스트를 모두 리뷰)

## 5대 축 리뷰 (The Five-Axis Review)

모든 리뷰는 다음 차원들에서 코드를 평가합니다:

### 1. 정확성 (Correctness)

코드가 의도한 대로 작동합니까?

- 스펙이나 태스크 요구 사항과 일치합니까?
- 예외 케이스(null, 비어 있음, 경계값)가 처리되었습니까?
- 해피 패스(happy path)뿐만 아니라 에러 경로도 처리되었습니까?
- 모든 테스트를 통과합니까? 테스트가 실제로 올바른 것을 테스트하고 있습니까?
- 오프 바이 원(off-by-one) 에러, 레이스 컨디션(race conditions), 또는 상태 불일치가 있습니까?

### 2. 가독성 및 단순성 (Readability & Simplicity)

다른 엔지니어(또는 에이전트)가 작성자의 설명 없이도 이 코드를 이해할 수 있습니까?

- 이름이 설명적이며 프로젝트 관례와 일치합니까? (컨텍스트 없는 `temp`, `data`, `result` 지양)
- 제어 흐름이 명확합니까? (중첩된 삼항 연산자, 깊은 콜백 지양)
- 코드가 논리적으로 구성되어 있습니까? (관련 코드 그룹화, 명확한 모듈 경계)
- 단순화해야 할 "기교"가 있습니까?
- **더 적은 라인으로 가능합니까?** (100줄로 충분할 것을 1000줄로 만든 것은 실패입니다.)
- **추상화가 그 복잡성만큼의 가치를 합니까?** (세 번째 유즈케이스가 생기기 전까지는 일반화하지 마세요.)
- 주석이 명확하지 않은 의도를 설명하는 데 도움이 됩니까? (뻔한 코드에는 주석을 달지 마세요.)
- 죽은 코드의 흔적이 있습니까? (사용되지 않는 변수(`_unused`), 하위 호환용 심(shims), 또는 `// 제거됨` 주석)

### 3. 아키텍처 (Architecture)

변경 사항이 시스템 설계와 잘 어울립니까?

- 기존 패턴을 따릅니까, 아니면 새로운 패턴을 도입합니까? 새롭다면 정당합니까?
- 깔끔한 모듈 경계를 유지합니까?
- 공유되어야 할 코드 중복이 있습니까?
- 의존성이 올바른 방향으로 흐릅니까? (순환 의존성 없음)
- 추상화 수준이 적절합니까? (과잉 설계되지 않았고, 너무 결합되지 않았는가)

### 4. 보안 (Security)

상세한 보안 가이드는 `security-and-hardening.ko.md`를 참조하세요. 변경 사항이 취약점을 도입합니까?

- 사용자 입력이 검증되고 정제(sanitize)되었습니까?
- 비밀 정보가 코드, 로그, 버전 관리 시스템에 노출되지 않았습니까?
- 필요한 곳에 인증(authentication)/인가(authorization) 체크가 있습니까?
- SQL 쿼리가 파라미터화되었습니까? (문자열 결합 금지)
- XSS를 방지하기 위해 출력이 인코딩되었습니까?
- 의존성이 알려진 취약점이 없는 신뢰할 수 있는 소스에서 왔습니까?
- 외부 소스(API, 로그, 사용자 콘텐츠, 설정 파일)의 데이터를 신뢰할 수 없는 것으로 취급합니까?
- 로직이나 렌더링에 사용하기 전에 시스템 경계에서 외부 데이터 흐름을 검증합니까?

### 5. 성능 (Performance)

상세한 프로파일링 및 최적화는 `performance-optimization.ko.md`를 참조하세요. 변경 사항이 성능 문제를 일으킵니까?

- N+1 쿼리 패턴이 있습니까?
- 제한 없는 루프나 제약 없는 데이터 페칭이 있습니까?
- 비동기여야 할 작업이 동기(sync)로 처리되고 있습니까?
- UI 컴포넌트에서 불필요한 리렌더링이 발생합니까?
- 목록 엔드포인트에 페이지네이션이 누락되었습니까?
- 핫 패스(hot paths)에서 큰 객체가 생성되고 있습니까?

## 변경 규모 (Change Sizing)

작고 집중된 변경이 리뷰하기 쉽고, 머지가 빠르며, 배포하기 안전합니다. 다음 규모를 목표로 하세요:

```
~100줄 변경   → 좋음. 한 번에 리뷰 가능.
~300줄 변경   → 하나의 논리적 변경이라면 수용 가능.
~1000줄 변경  → 너무 큼. 분할하세요.
```

**"하나의 변경"의 정의:** 그 자체로 완결된 수정을 의미하며, 하나의 사항을 다루고, 관련 테스트를 포함하며, 제출 후에도 시스템을 기능적으로 유지하는 단위입니다. 기능 전체가 아니라 기능의 한 부분이어야 합니다.

**변경 사항이 너무 클 때의 분할 전략:**

| 전략 | 방법 | 시기 |
|----------|-----|------|
| **스택 (Stack)** | 작은 변경을 제출하고, 이를 기반으로 다음 작업 시작 | 순차적 의존성이 있을 때 |
| **파일 그룹별** | 리뷰어가 다른 그룹별로 변경 사항 분리 | 횡단 관심사(cross-cutting concerns)일 때 |
| **수평적** | 공유 코드/스텁을 먼저 만들고, 나중에 소비자 구현 | 레이어드 아키텍처일 때 |
| **수직적** | 기능을 더 작은 풀스택 슬라이스로 분할 | 기능 작업일 때 |

**대규모 변경이 허용되는 경우:** 전체 파일 삭제나 리뷰어가 모든 라인이 아닌 의도만 확인하면 되는 자동화된 리팩토링의 경우.

**리팩토링과 기능 작업을 분리하세요.** 기존 코드를 리팩토링하면서 새로운 동작을 추가하는 것은 두 개의 변경 사항입니다. 이들을 별도로 제출하세요. 작은 정리 작업(변수명 변경 등)은 리뷰어의 재량에 따라 기능 커밋에 포함될 수 있습니다.

## 변경 사항 설명 (Change Descriptions)

모든 변경 사항은 버전 관리 히스토리에서 독립적으로 이해될 수 있는 설명을 가져야 합니다.

**첫 줄:** 짧고, 명령조이며, 독립적이어야 합니다. "FizzBuzz RPC를 삭제함" (O), "FizzBuzz RPC를 삭제하는 중" (X). 히스토리를 검색하는 사람이 diff를 읽지 않고도 변경 내용을 이해할 수 있을 만큼 정보가 충분해야 합니다.

**본문:** 무엇이 왜 변하는지 설명하세요. 코드 자체에서는 보이지 않는 컨텍스트, 결정 사항, 추론을 포함하세요. 관련 버그 번호, 벤치마크 결과, 또는 설계 문서 링크를 포함하세요. 접근 방식의 한계가 있다면 이를 인정하세요.

**안티 패턴:** "버그 수정", "빌드 수정", "패치 추가", "코드를 A에서 B로 옮김", "1단계", "편의 함수 추가".

## 리뷰 프로세스 (Review Process)

### 1단계: 컨텍스트 이해

코드를 보기 전에 의도를 이해하세요:

```
- 이 변경을 통해 무엇을 달성하려고 하는가?
- 어떤 스펙이나 태스크를 구현하는가?
- 예상되는 동작 변경은 무엇인가?
```

### 2단계: 테스트를 먼저 리뷰하기

테스트는 의도와 커버리지를 드러냅니다:

```
- 변경 사항에 대한 테스트가 존재하는가?
- 구현 세부 사항이 아닌 동작(behavior)을 테스트하는가?
- 예외 케이스가 커버되었는가?
- 테스트 이름이 설명적인가?
- 코드가 바뀌었을 때 테스트가 회귀를 잡아낼 수 있는가?
```

### 3단계: 구현 리뷰하기

다섯 가지 축을 염두에 두고 코드를 훑어보세요:

```
수정된 각 파일에 대해:
1. 정확성: 이 코드가 테스트에서 말하는 대로 작동하는가?
2. 가독성: 도움 없이도 이 코드를 이해할 수 있는가?
3. 아키텍처: 시스템과 잘 어울리는가?
4. 보안: 취약점은 없는가?
5. 성능: 병목 현상은 없는가?
```

### 4단계: 발견 사항 분류

작성자가 필수 사항과 선택 사항을 구분할 수 있도록 모든 코멘트에 심각도 라벨을 붙이세요:

| 접두사 | 의미 | 작성자 행동 |
|--------|---------|---------------|
| *(접두사 없음)* | 필수 변경 사항 | 머지 전에 반드시 수정해야 함 |
| **Critical:** | 머지 차단 요소 | 보안 취약점, 데이터 손실, 기능 마비 |
| **Nit:** | 사소함, 선택 사항 | 작성자가 무시 가능 — 포맷팅, 스타일 선호도 |
| **Optional:** / **Consider:** | 제안 사항 | 고려해 볼 가치가 있지만 필수는 아님 |
| **FYI** | 정보 제공용 | 조치 불필요 — 미래를 위한 참고 컨텍스트 |

이렇게 하면 작성자가 모든 피드백을 필수로 오해하여 선택적 제안에 시간을 낭비하는 것을 방지할 수 있습니다.

### 5단계: 검증 내용 확인

작성자의 검증 내용을 확인하세요:

```
- 어떤 테스트를 실행했는가?
- 빌드가 성공했는가?
- 변경 사항을 수동으로 테스트했는가?
- UI 변경의 경우 스크린샷이 있는가?
- 전후(before/after) 비교가 있는가?
```

## 멀티 모델 리뷰 패턴 (Multi-Model Review Pattern)

다양한 리뷰 관점을 위해 서로 다른 모델을 사용하세요:

```
모델 A가 코드를 작성함
    │
    ▼
모델 B가 정확성과 아키텍처를 리뷰함
    │
    ▼
모델 A가 피드백을 반영함
    │
    ▼
인간이 최종 결정을 내림
```

이렇게 하면 단일 모델이 놓칠 수 있는 이슈를 잡아낼 수 있습니다. 모델마다 사각지대가 다르기 때문입니다.

**리뷰 에이전트를 위한 프롬프트 예시:**
```
이 코드 변경 사항의 정확성, 보안, 그리고 프로젝트 관례 준수 여부를
리뷰해줘. 스펙은 [X]이고, 변경 사항은 [Y]를 수행해야 해.
이슈가 있다면 Critical, Important, Suggestion으로 표시해줘.
```

## 죽은 코드 관리 (Dead Code Hygiene)

리팩토링이나 구현 변경 후에는 고립된 코드가 있는지 확인하세요:

1. 이제 도달할 수 없거나 사용되지 않는 코드를 식별합니다.
2. 이를 명시적으로 나열합니다.
3. **삭제하기 전에 물어보세요:** "이제 사용되지 않는 이 요소들을 삭제할까요? [목록]"

죽은 코드를 방치하지 마세요. 미래의 독자와 에이전트를 혼란스럽게 만듭니다. 하지만 확신이 없는 것을 무단으로 삭제하지는 마세요. 의심스러우면 물어보세요.

```
죽은 코드 식별됨:
- src/utils/date.ts의 formatLegacyDate() — formatDate()로 대체됨
- src/components/의 OldTaskCard 컴포넌트 — TaskCard로 대체됨
- src/config.ts의 LEGACY_API_URL 상수 — 더 이상 참조 없음
→ 삭제해도 될까요?
```

## 리뷰 속도 (Review Speed)

느린 리뷰는 전체 팀의 발목을 잡습니다. 리뷰를 위해 컨텍스트를 전환하는 비용은 다른 사람들에게 부과되는 대기 비용보다 작습니다.

- **영업일 기준 1일 이내에 응답하세요.** 이것은 마지노선이지 목표가 아닙니다.
- **이상적인 속도:** 리뷰 요청이 오면 얼마 지나지 않아 응답하세요(집중 코딩 중이 아니라면). 일반적인 변경 사항은 하루 안에 여러 번의 리뷰 라운드를 마칠 수 있어야 합니다.
- **빠른 피드백 우선:** 최종 승인을 빨리하는 것보다 개별 응답을 빨리하는 것이 중요합니다. 여러 라운드가 필요하더라도 빠른 피드백은 작성자의 답답함을 줄여줍니다.
- **대규모 변경:** 한꺼번에 리뷰하기보다는 작성자에게 분할을 요청하세요.

## 의견 불일치 처리 (Handling Disagreements)

리뷰 분쟁을 해결할 때는 다음 우선순위를 적용하세요:

1. **기술적 사실과 데이터**가 의견이나 선호도보다 우선합니다.
2. **스타일 가이드**가 스타일 문제의 절대적인 권위자입니다.
3. **소프트웨어 설계**는 개인적 취향이 아닌 엔지니어링 원칙에 따라 평가되어야 합니다.
4. **코드베이스 일관성**은 전체적인 건강성을 해치지 않는 한 수용 가능합니다.

**"나중에 정리할게요"를 받아들이지 마세요.** 경험상 미뤄진 정리는 거의 일어나지 않습니다. 정말 긴급한 상황이 아니라면 제출 전에 정리를 요구하세요. 주변 문제를 이번 변경에서 해결할 수 없다면, 스스로를 담당자로 지정하여 버그 티켓을 발행하게 하세요.

## 리뷰의 정직함 (Honesty in Review)

본인, 다른 에이전트, 또는 인간이 작성한 코드를 리뷰할 때:

- **무조건 승인하지 마세요.** 근거 없는 "LGTM"은 아무에게도 도움이 되지 않습니다.
- **진짜 이슈를 완곡하게 표현하지 마세요.** 프로덕션에 영향을 줄 버그인데 "약간의 우려가 있네요"라고 하는 것은 정직하지 못한 것입니다.
- **가능하다면 문제를 정량화하세요.** "이 N+1 쿼리는 목록의 항목당 ~50ms를 추가할 것입니다"가 "느려질 수 있습니다"보다 낫습니다.
- **분명한 문제가 있는 접근 방식은 반박하세요.** 아첨은 리뷰 실패의 원인입니다. 구현에 문제가 있다면 직접적으로 말하고 대안을 제시하세요.
- **상대가 결정권을 가졌다면 우아하게 승복하세요.** 작성자가 전체 컨텍스트를 가지고 있고 동의하지 않는다면, 그의 판단에 맡기세요. 사람이 아닌 코드에 대해 코멘트하세요. 개인적인 비판이 아닌 코드 자체에 집중하세요.

## 의존성 관리 (Dependency Discipline)

코드 리뷰의 일부는 의존성 리뷰입니다:

**의존성을 추가하기 전에:**
1. 기존 스택으로 해결 가능한가? (대부분 가능합니다.)
2. 의존성의 크기는 얼마인가? (번들 영향을 확인하세요.)
3. 활발히 유지보수되는가? (마지막 커밋, 열린 이슈 확인.)
4. 알려진 취약점이 있는가? (`npm audit`)
5. 라이선스가 무엇인가? (프로젝트와 호환되어야 함.)

**규칙:** 새로운 의존성보다 표준 라이브러리와 기존 유틸리티를 선호하세요. 모든 의존성은 부채입니다.

## 리뷰 체크리스트 (The Review Checklist)

```markdown
## 리뷰: [PR/변경 제목]

### 컨텍스트 (Context)
- [ ] 이 변경이 무엇을 하고 왜 하는지 이해함

### 정확성 (Correctness)
- [ ] 변경 사항이 스펙/태스크 요구 사항과 일치함
- [ ] 예외 케이스 처리됨
- [ ] 에러 경로 처리됨
- [ ] 테스트가 변경 사항을 충분히 커버함

### 가독성 (Readability)
- [ ] 이름이 명확하고 일관됨
- [ ] 로직이 명료함
- [ ] 불필요한 복잡성 없음

### 아키텍처 (Architecture)
- [ ] 기존 패턴을 따름
- [ ] 불필요한 결합이나 의존성 없음
- [ ] 추상화 수준이 적절함

### 보안 (Security)
- [ ] 코드에 비밀 정보 없음
- [ ] 경계에서 입력 검증됨
- [ ] 주입(injection) 취약점 없음
- [ ] 인증/인가 체크 완료
- [ ] 외부 데이터 소스를 신뢰할 수 없는 것으로 취급함

### 성능 (Performance)
- [ ] N+1 패턴 없음
- [ ] 제한 없는 작업 없음
- [ ] 목록 엔드포인트에 페이지네이션 있음

### 검증 (Verification)
- [ ] 테스트 통과
- [ ] 빌드 성공
- [ ] 수동 검증 완료 (해당하는 경우)

### 판정 (Verdict)
- [ ] **승인 (Approve)** — 머지 준비 완료
- [ ] **수정 요청 (Request changes)** — 이슈 수정 필요
```

## 흔한 자기합리화 (Common Rationalizations)

| 자기합리화 | 실제 상황 |
|---|---|
| "작동하니까 이 정도면 충분해요" | 읽기 어렵고, 불안전하며, 아키텍처가 틀린 코드는 나중에 더 큰 부채가 됩니다. |
| "내가 썼으니까 맞을 거예요" | 작성자는 자신의 가정을 보지 못합니다. 모든 변경은 다른 사람의 눈을 거칠 때 가치가 높아집니다. |
| "나중에 정리할게요" | '나중'은 오지 않습니다. 리뷰가 품질 게이트입니다. 머지 전이 아니라 머지 후에 정리를 요구하세요. |
| "AI가 생성한 거니까 괜찮겠죠" | AI 코드는 더 꼼꼼히 봐야 합니다. 틀린 내용도 그럴듯하고 자신 있게 제시하기 때문입니다. |
| "테스트가 통과했으니 됐어요" | 테스트는 필요조건이지 충분조건이 아닙니다. 아키텍처 문제, 보안 이슈, 가독성 우려는 잡지 못합니다. |

## 위험 신호 (Red Flags)

- 리뷰 없이 머지된 PR
- 테스트 통과 여부만 확인하는 리뷰 (다른 축 무시)
- 실제 리뷰 근거 없는 "LGTM"
- 보안 관련 변경 사항에 보안 중심 리뷰가 없음
- "리뷰하기에 너무 큰" 대규모 PR (분할 필요)
- 버그 수정 PR에 회귀 테스트가 없음
- 심각도 라벨 없는 리뷰 코멘트 — 무엇이 필수인지 알기 어려움
- "나중에 고칠게요"를 수용함 — 절대 일어나지 않음

## 검증 (Verification)

리뷰 완료 후:

- [ ] 모든 Critical 이슈가 해결되었는가
- [ ] 모든 Important 이슈가 해결되었거나 정당한 사유와 함께 명시적으로 미루어졌는가
- [ ] 테스트가 통과하는가
- [ ] 빌드가 성공하는가
- [ ] 검증 내용이 문서화되었는가 (무엇이 바뀌었고 어떻게 검증되었는지)

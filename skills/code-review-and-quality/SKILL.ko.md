# Code Review and Quality

## Overview

품질 게이트를 갖춘 다차원 코드 리뷰. 모든 변경은 merge 전에 리뷰됩니다 — 예외 없음. 리뷰는 다섯 축을 다룹니다: 정확성, 가독성, 아키텍처, 보안, 성능.

**승인 기준:** 완벽하지 않더라도, 변경이 전반적인 코드 건강성을 분명히 향상시키면 승인하세요. 완벽한 코드는 존재하지 않습니다 — 목표는 지속적 개선입니다. 자신이 작성했을 방식과 정확히 같지 않다고 변경을 막지 마세요. 코드베이스를 향상시키고 프로젝트 규약을 따른다면, 승인하세요.

## When to Use

- 모든 PR이나 변경을 merge하기 전에
- 기능 구현을 완료한 후
- 다른 agent나 모델이 평가해야 할 코드를 생성했을 때
- 기존 코드를 리팩터링할 때
- 모든 버그 수정 후 (수정과 회귀 테스트 둘 다 리뷰)

## 5축 리뷰

모든 리뷰는 이 차원들에서 코드를 평가합니다:

### 1. 정확성 (Correctness)

코드가 주장하는 것을 하는가?

- spec이나 task 요구사항과 일치하는가?
- 엣지 케이스가 처리되는가 (null, 빈 값, 경계값)?
- 에러 경로가 처리되는가 (happy path만이 아니라)?
- 모든 테스트를 통과하는가? 테스트가 실제로 올바른 것을 테스트하는가?
- off-by-one 에러, race condition, 또는 상태 불일치가 있는가?

### 2. 가독성 & 단순성 (Readability & Simplicity)

다른 엔지니어(또는 agent)가 작성자의 설명 없이 이 코드를 이해할 수 있는가?

- 이름이 서술적이고 프로젝트 규약과 일관적인가? (context 없는 `temp`, `data`, `result` 없음)
- 제어 흐름이 단순한가 (중첩 ternary, 깊은 callback 회피)?
- 코드가 논리적으로 조직되어 있는가 (관련 코드 그룹화, 명확한 모듈 경계)?
- 단순화되어야 할 "영리한" 트릭이 있는가?
- **더 적은 줄로 할 수 있는가?** (100줄로 충분한데 1000줄이면 실패)
- **추상화가 그 복잡도만큼 값을 하는가?** (세 번째 사용 사례까지 일반화하지 말 것)
- 주석이 명백하지 않은 의도를 명확히 하는 데 도움이 될까? (하지만 명백한 코드에 주석 달지 말 것.)
- 죽은 코드 잔재가 있는가: no-op 변수(`_unused`), 하위 호환 shim, 또는 `// removed` 주석?

### 3. 아키텍처 (Architecture)

변경이 시스템의 설계에 맞는가?

- 기존 패턴을 따르는가, 아니면 새 패턴을 도입하는가? 새것이라면 정당화되는가?
- 깨끗한 모듈 경계를 유지하는가?
- 공유되어야 할 코드 중복이 있는가?
- 의존성이 올바른 방향으로 흐르는가 (순환 의존성 없음)?
- 추상화 수준이 적절한가 (과도하게 엔지니어링되지 않고, 너무 결합되지 않음)?

### 4. 보안 (Security)

상세한 보안 가이드는 `security-and-hardening`을 참고하세요. 변경이 취약점을 도입하는가?

- 사용자 입력이 검증되고 sanitize되는가?
- secrets가 코드, 로그, version control 밖에 유지되는가?
- 필요한 곳에서 인증/인가가 검사되는가?
- SQL 쿼리가 parameterize되는가 (문자열 연결 없음)?
- 출력이 XSS 방지를 위해 encode되는가?
- 의존성이 알려진 취약점 없는 신뢰할 수 있는 출처에서 오는가?
- 외부 출처(API, 로그, 사용자 콘텐츠, config 파일)의 데이터가 신뢰할 수 없는 것으로 취급되는가?
- 외부 데이터 흐름이 로직이나 렌더링에 사용되기 전 시스템 경계에서 검증되는가?

### 5. 성능 (Performance)

상세한 profiling과 최적화는 `performance-optimization`을 참고하세요. 변경이 성능 문제를 도입하는가?

- N+1 쿼리 패턴이 있는가?
- 무한 루프나 제약 없는 데이터 fetch가 있는가?
- async여야 할 동기 작업이 있는가?
- UI 컴포넌트에 불필요한 re-render가 있는가?
- 목록 endpoint에 pagination이 빠져 있는가?
- hot path에서 생성되는 큰 객체가 있는가?

## 변경 크기 조정

작고 집중된 변경은 리뷰하기 쉽고, merge가 빠르고, 배포가 안전합니다. 이 크기를 목표로 하세요:

```
~100줄 변경   → 좋음. 한 번에 리뷰 가능.
~300줄 변경   → 단일 논리적 변경이라면 허용 가능.
~1000줄 변경  → 너무 큼. 분할하세요.
```

**"한 변경"으로 치는 것:** 한 가지를 다루고, 관련 테스트를 포함하고, 제출 후 시스템을 기능하게 유지하는 단일 자기 완결적 수정. 전체 기능이 아니라 기능의 한 부분.

**변경이 너무 클 때 분할 전략:**

| 전략 | 어떻게 | 언제 |
|----------|-----|------|
| **Stack** | 작은 변경을 제출하고, 그 위에 다음을 시작 | 순차 의존성 |
| **파일 그룹별** | 다른 리뷰어가 필요한 그룹별로 변경 분리 | 횡단 관심사 |
| **수평** | 공유 코드/stub을 먼저 만들고, 그 다음 소비자 | 계층 아키텍처 |
| **수직** | 기능의 더 작은 full-stack 슬라이스로 분할 | 기능 작업 |

**큰 변경이 허용될 때:** 리뷰어가 모든 줄이 아니라 의도만 검증하면 되는 전체 파일 삭제와 자동 리팩터링.

**리팩터링을 기능 작업에서 분리하세요.** 기존 코드를 리팩터링하면서 새 동작을 추가하는 변경은 두 개의 변경입니다 — 별도로 제출하세요. 작은 정리(변수 이름 변경)는 리뷰어 재량으로 포함할 수 있습니다.

## 변경 설명

모든 변경에는 version control 히스토리에서 단독으로 서는 설명이 필요합니다.

**첫 줄:** 짧고, 명령형이고, 단독으로 섬. "Deleting the FizzBuzz RPC"가 아니라 "Delete the FizzBuzz RPC." 히스토리를 검색하는 사람이 diff를 읽지 않고도 변경을 이해할 만큼 충분히 정보가 있어야 함.

**본문:** 무엇이 왜 바뀌는지. 코드 자체에 보이지 않는 context, 결정, 추론을 포함. 관련 있는 곳에 bug 번호, 벤치마크 결과, 또는 설계 문서를 링크. 접근의 단점이 있을 때 인정.

**안티패턴:** "Fix bug," "Fix build," "Add patch," "Moving code from A to B," "Phase 1," "Add convenience functions."

## 리뷰 프로세스

### Step 1: Context 이해

코드를 보기 전에, 의도를 이해하세요:

```
- 이 변경이 무엇을 달성하려는가?
- 어떤 spec이나 task를 구현하는가?
- 예상되는 동작 변경은 무엇인가?
```

### Step 2: 테스트를 먼저 리뷰

테스트는 의도와 커버리지를 드러냅니다:

```
- 변경에 대한 테스트가 존재하는가?
- 동작을 테스트하는가 (구현 세부사항이 아니라)?
- 엣지 케이스가 커버되는가?
- 테스트에 서술적 이름이 있는가?
- 코드가 바뀌면 테스트가 회귀를 잡을까?
```

### Step 3: 구현 리뷰

다섯 축을 염두에 두고 코드를 훑으세요:

```
변경된 각 파일에 대해:
1. 정확성: 이 코드가 테스트가 말하는 것을 하는가?
2. 가독성: 도움 없이 이해할 수 있는가?
3. 아키텍처: 이것이 시스템에 맞는가?
4. 보안: 취약점이 있는가?
5. 성능: 병목이 있는가?
```

### Step 4: 발견 분류

모든 코멘트에 심각도를 레이블해 작성자가 필수 vs 선택을 알게 하세요:

| 접두 | 의미 | 작성자 조치 |
|--------|---------|---------------|
| *(접두 없음)* | 필수 변경 | merge 전 반드시 해결 |
| **Critical:** | merge 차단 | 보안 취약점, 데이터 손실, 깨진 기능 |
| **Nit:** | 사소, 선택 | 작성자가 무시 가능 — 포맷팅, 스타일 선호 |
| **Optional:** / **Consider:** | 제안 | 고려할 가치 있으나 필수 아님 |
| **FYI** | 정보 제공만 | 조치 불필요 — 향후 참고용 context |

이는 작성자가 모든 피드백을 필수로 취급하고 선택적 제안에 시간을 낭비하는 것을 방지합니다.

### Step 5: 검증을 검증

작성자의 검증 스토리를 확인하세요:

```
- 어떤 테스트가 실행되었나?
- 빌드가 통과했나?
- 변경이 수동으로 테스트되었나?
- UI 변경에 스크린샷이 있나?
- before/after 비교가 있나?
```

## 다중 모델 리뷰 패턴

다른 리뷰 관점에 다른 모델을 사용하세요:

```
모델 A가 코드 작성
    │
    ▼
모델 B가 정확성과 아키텍처를 리뷰
    │
    ▼
모델 A가 피드백 해결
    │
    ▼
사람이 최종 결정
```

이는 단일 모델이 놓칠 수 있는 이슈를 잡습니다 — 다른 모델은 다른 맹점을 가집니다.

**리뷰 agent용 예시 프롬프트:**
```
Review this code change for correctness, security, and adherence to
our project conventions. The spec says [X]. The change should [Y].
Flag any issues as Critical, Important, or Suggestion.
```

## 죽은 코드 위생

모든 리팩터링이나 구현 변경 후, 고아 코드를 확인하세요:

1. 이제 도달 불가능하거나 사용되지 않는 코드 식별
2. 명시적으로 나열
3. **삭제 전 물어보기:** "이제 사용되지 않는 이 요소들을 제거할까요: [목록]?"

죽은 코드를 방치하지 마세요 — 미래 독자와 agent를 혼란스럽게 합니다. 하지만 확신하지 못하는 것을 조용히 삭제하지 마세요. 의심스러우면 물어보세요.

```
DEAD CODE IDENTIFIED:
- formatLegacyDate() in src/utils/date.ts — replaced by formatDate()
- OldTaskCard component in src/components/ — replaced by TaskCard
- LEGACY_API_URL constant in src/config.ts — no remaining references
→ Safe to remove these?
```

## 리뷰 속도

느린 리뷰는 팀 전체를 막습니다. 리뷰하러 context를 전환하는 비용은 다른 사람에게 부과되는 대기 비용보다 작습니다.

- **영업일 하루 이내 응답** — 이것은 목표가 아니라 최대치
- **이상적 cadence:** 집중 코딩에 깊이 들어가 있지 않다면, 리뷰 요청이 도착한 직후 응답. 일반적인 변경은 하루에 여러 리뷰 라운드를 완료해야 함
- **빠른 최종 승인보다 빠른 개별 응답을 우선**. 여러 라운드가 필요해도 빠른 피드백은 좌절을 줄임
- **큰 변경:** 거대한 changeset 하나를 리뷰하기보다 작성자에게 분할을 요청

## 의견 불일치 처리

리뷰 분쟁을 해결할 때, 이 위계를 적용하세요:

1. **기술적 사실과 데이터**가 의견과 선호를 압도
2. **스타일 가이드**가 스타일 문제의 절대 권위
3. **소프트웨어 설계**는 개인 선호가 아니라 엔지니어링 원칙으로 평가
4. **코드베이스 일관성**은 전반적 건강성을 저해하지 않으면 허용 가능

**"나중에 정리할게"를 받아들이지 마세요.** 경험상 미뤄진 정리는 거의 일어나지 않습니다. 진짜 비상이 아니라면 제출 전 정리를 요구하세요. 주변 이슈를 이 변경에서 다룰 수 없다면, 자기 할당으로 bug를 제출하도록 요구하세요.

## 리뷰에서의 정직

코드를 리뷰할 때 — 자신, 다른 agent, 또는 사람이 작성한 것이든:

- **고무 도장을 찍지 마세요.** 리뷰 증거 없는 "LGTM"은 누구에게도 도움이 안 됩니다.
- **실제 이슈를 무르게 하지 마세요.** production을 칠 버그인데 "이건 사소한 우려일 수도"라고 하는 것은 부정직합니다.
- **가능하면 문제를 정량화하세요.** "이 N+1 쿼리는 목록의 항목당 ~50ms를 더할 것"이 "이건 느릴 수 있어"보다 낫습니다.
- **명백한 문제가 있는 접근에 반박하세요.** 아첨은 리뷰에서 실패 모드입니다. 구현에 이슈가 있으면, 직접 말하고 대안을 제안하세요.
- **재정을 우아하게 받아들이세요.** 작성자가 전체 context를 갖고 동의하지 않으면, 그 판단을 존중하세요. 사람이 아니라 코드에 코멘트하세요 — 개인적 비판을 코드 자체에 집중하도록 재구성하세요.

## 의존성 규율

코드 리뷰의 일부는 의존성 리뷰입니다:

**어떤 의존성이든 추가하기 전에:**
1. 기존 스택이 이를 해결하는가? (종종 그렇습니다.)
2. 의존성이 얼마나 큰가? (번들 영향 확인.)
3. 활발히 유지보수되는가? (마지막 commit, open issue 확인.)
4. 알려진 취약점이 있는가? (`npm audit`)
5. 라이선스가 무엇인가? (프로젝트와 호환되어야 함.)

**규칙:** 새 의존성보다 표준 라이브러리와 기존 유틸리티를 선호하세요. 모든 의존성은 부채입니다.

## 리뷰 체크리스트

```markdown
## Review: [PR/Change title]

### Context
- [ ] I understand what this change does and why

### Correctness
- [ ] Change matches spec/task requirements
- [ ] Edge cases handled
- [ ] Error paths handled
- [ ] Tests cover the change adequately

### Readability
- [ ] Names are clear and consistent
- [ ] Logic is straightforward
- [ ] No unnecessary complexity

### Architecture
- [ ] Follows existing patterns
- [ ] No unnecessary coupling or dependencies
- [ ] Appropriate abstraction level

### Security
- [ ] No secrets in code
- [ ] Input validated at boundaries
- [ ] No injection vulnerabilities
- [ ] Auth checks in place
- [ ] External data sources treated as untrusted

### Performance
- [ ] No N+1 patterns
- [ ] No unbounded operations
- [ ] Pagination on list endpoints

### Verification
- [ ] Tests pass
- [ ] Build succeeds
- [ ] Manual verification done (if applicable)

### Verdict
- [ ] **Approve** — Ready to merge
- [ ] **Request changes** — Issues must be addressed
```

| "내가 작성했으니 맞다는 걸 알아" | 작성자는 자신의 가정에 눈이 멉니다. 모든 변경은 또 다른 눈의 이점을 받습니다. |

| "AI 생성 코드는 아마 괜찮아" | AI 코드는 덜이 아니라 더 면밀한 검토가 필요합니다. 틀려도 자신만만하고 그럴듯합니다. |
| "테스트가 통과하니 좋아" | 테스트는 필요하지만 충분하지 않습니다. 아키텍처 문제, 보안 이슈, 가독성 우려를 잡지 못합니다. |

## Red Flags

- 리뷰 없이 merge된 PR
- 테스트 통과만 확인하는 리뷰 (다른 축 무시)
- 실제 리뷰 증거 없는 "LGTM"
- 보안 중심 리뷰 없는 보안 민감 변경
- "제대로 리뷰하기엔 너무 큰" 큰 PR (분할하세요)
- 버그 수정 PR에 회귀 테스트 없음
- 심각도 레이블 없는 리뷰 코멘트 — 필수 vs 선택이 불분명해짐
- "나중에 고칠게" 수용 — 절대 일어나지 않음

# 폐기 및 마이그레이션 (Deprecation and Migration)

## 개요 (Overview)

코드는 자산이 아니라 부채입니다. 모든 코드 라인에는 버그 수정, 의존성 업데이트, 보안 패치 적용, 새로운 엔지니어 온보딩 등 지속적인 유지보수 비용이 발생합니다. 폐기(Deprecation)는 더 이상 제 역할을 하지 못하는 코드를 제거하는 규율이며, 마이그레이션(Migration)은 사용자를 예전 것에서 새로운 것으로 안전하게 이동시키는 프로세스입니다.

대부분의 엔지니어링 조직은 무언가를 만드는 데는 능숙하지만, 그것을 제거하는 데는 서툽니다. 이 스킬은 그 간극을 메웁니다.

## 사용 시기 (When to Use)

- 오래된 시스템, API 또는 라이브러리를 새로운 것으로 교체할 때
- 더 이상 필요하지 않은 기능을 일몰(sunsetting)시킬 때
- 중복된 구현을 통합할 때
- 아무도 소유하지 않지만 모두가 의존하는 죽은 코드(dead code)를 제거할 때
- 새로운 시스템의 수명 주기를 계획할 때 (폐기 계획은 설계 시점부터 시작됩니다)
- 레거시 시스템을 유지할지 아니면 마이그레이션에 투자할지 결정할 때

## 핵심 원칙 (Core Principles)

### 코드는 부채입니다 (Code Is a Liability)

모든 코드 라인에는 지속적인 비용이 따릅니다. 테스트, 문서화, 보안 패치, 의존성 업데이트가 필요하며 근처에서 작업하는 모든 이에게 인지적 오버헤드를 발생시킵니다. 코드의 가치는 코드가 제공하는 기능에 있지, 코드 그 자체에 있지 않습니다. 동일한 기능을 더 적은 코드, 더 낮은 복잡성, 혹은 더 나은 추상화로 제공할 수 있다면 예전 코드는 사라져야 합니다.

### 하이럼의 법칙(Hyrum's Law)이 제거를 어렵게 만듭니다

사용자가 충분히 많아지면 버그, 타이밍 특성, 문서화되지 않은 부작용을 포함하여 관찰 가능한 모든 동작에 누군가는 의존하게 됩니다. 이것이 폐기에 단순한 공지가 아닌 능동적인 마이그레이션이 필요한 이유입니다. 대체제가 복제하지 못하는 동작에 의존하고 있는 사용자는 "그냥 바꿀" 수 없습니다.

### 폐기 계획은 설계 시점부터 시작됩니다

새로운 것을 만들 때 자문해 보세요. "3년 후에 이것을 어떻게 제거할 것인가?" 깔끔한 인터페이스, 피처 플래그(feature flags), 최소한의 노출 영역으로 설계된 시스템은 구현 세부 사항을 사방에 노출하는 시스템보다 폐기하기 쉽습니다.

## 폐기 결정 (The Deprecation Decision)

무엇인가를 폐기하기 전에 다음 질문에 답하세요:

```
1. 이 시스템이 여전히 고유한 가치를 제공하는가?
   → 그렇다면 유지하세요. 아니라면 계속 진행하세요.

2. 얼마나 많은 사용자/소비자가 여기에 의존하고 있는가?
   → 마이그레이션 범위를 정량화하세요.

3. 대체제가 존재하는가?
   → 없다면 대체제를 먼저 만드세요. 대안 없이 폐기하지 마세요.

4. 각 소비자별 마이그레이션 비용은 얼마인가?
   → 자동화가 매우 쉽다면 직접 수행하세요. 수동이며 많은 노력이 든다면 유지보수 비용과 비교해 보세요.

5. 폐기하지 않을 경우 발생하는 지속적인 유지보수 비용은 얼마인가?
   → 보안 위험, 엔지니어 시간, 복잡성으로 인한 기회비용 등.
```

## 강제적(Compulsory) 폐기 vs 권고적(Advisory) 폐기

| 유형 | 사용 시기 | 메커니즘 |
|------|-------------|-----------|
| **권고적 (Advisory)** | 마이그레이션이 선택사항이며, 기존 시스템이 안정적임 | 경고, 문서화, 넛지(nudges). 사용자는 자신의 일정에 맞춰 마이그레이션함. |
| **강제적 (Compulsory)** | 기존 시스템에 보안 이슈가 있거나, 진행을 방해하거나, 유지보수 비용을 감당할 수 없음 | 확정된 마감일. 특정 날짜에 시스템이 제거됨. 마이그레이션 도구 제공 필수. |

**기본적으로 권고적 폐기를 지향하세요.** 유지보수 비용이나 위험이 마이그레이션을 강제할 만큼 정당할 때만 강제적 폐기를 사용하세요. 강제적 폐기에는 마이그레이션 도구, 문서화 및 지원 제공이 필수적입니다. 단순히 마감일만 공지해서는 안 됩니다.

## 마이그레이션 프로세스 (The Migration Process)

### 1단계: 대체제 구축

작동하는 대안 없이 폐기하지 마세요. 대체제는 다음을 충족해야 합니다:

- 기존 시스템의 모든 핵심 유즈케이스(use cases)를 커버함
- 문서화 및 마이그레이션 가이드가 있음
- 프로덕션 환경에서 입증됨 (단순히 "이론적으로 더 나은" 것이 아님)

### 2단계: 공지 및 문서화

```markdown
## 폐기 공지: OldService

**상태:** 2025-03-01부로 폐기됨
**대체제:** NewService (아래 마이그레이션 가이드 참조)
**제거 날짜:** 권고적 — 아직 확정된 마감일 없음
**사유:** OldService는 수동 스케일링이 필요하며 관측 가능성(observability)이 부족합니다.
            NewService는 이 두 가지를 모두 자동으로 처리합니다.

### 마이그레이션 가이드
1. `import { client } from 'old-service'`를 `import { client } from 'new-service'`로 교체
2. 설정 업데이트 (아래 예시 참조)
3. 마이그레이션 검증 스크립트 실행: `npx migrate-check`
```

### 3단계: 점진적 마이그레이션

소비자를 한꺼번에 옮기지 말고 하나씩 옮기세요. 각 소비자별로:

```
1. 폐기된 시스템과의 모든 접점 식별
2. 대체제를 사용하도록 업데이트
3. 동작이 일치하는지 검증 (테스트, 통합 체크)
4. 이전 시스템에 대한 참조 제거
5. 회귀(regressions)가 없는지 확인
```

**변경 책임의 원칙(The Churn Rule):** 폐기되는 인프라의 소유자라면, 사용자를 마이그레이션시키거나 마이그레이션이 필요 없는 하위 호환 업데이트를 제공할 책임이 있습니다. 폐기 공지만 하고 사용자에게 알아서 하라고 남겨두지 마세요.

### 4단계: 이전 시스템 제거

모든 소비자가 마이그레이션을 마친 후에만:

```
1. 활성 사용량이 0인지 확인 (메트릭, 로그, 의존성 분석)
2. 코드 제거
3. 관련 테스트, 문서 및 설정 제거
4. 폐기 공지 제거
5. 축하 — 코드를 제거하는 것은 훌륭한 성취입니다.
```

## 마이그레이션 패턴 (Migration Patterns)

### 스트랭글러 패턴 (Strangler Pattern)

이전 시스템과 새로운 시스템을 병렬로 실행합니다. 트래픽을 이전 시스템에서 새로운 시스템으로 점진적으로 라우팅합니다. 이전 시스템이 처리하는 트래픽이 0%가 되면 제거합니다.

```
1단계: 새로운 시스템 0%, 이전 시스템 100% 처리
2단계: 새로운 시스템 10% 처리 (카나리(canary))
3단계: 새로운 시스템 50% 처리
4단계: 새로운 시스템 100% 처리, 이전 시스템 대기
5단계: 이전 시스템 제거
```

### 어댑터 패턴 (Adapter Pattern)

이전 인터페이스의 호출을 새로운 구현으로 번역해주는 어댑터를 만듭니다. 백엔드를 마이그레이션하는 동안 소비자는 계속해서 이전 인터페이스를 사용할 수 있습니다.

```typescript
// 어댑터: 이전 인터페이스, 새로운 구현
class LegacyTaskService implements OldTaskAPI {
  constructor(private newService: NewTaskService) {}

  // 이전 메서드 시그니처, 새로운 구현으로 위임
  getTask(id: number): OldTask {
    const task = this.newService.findById(String(id));
    return this.toOldFormat(task);
  }
}
```

### 피처 플래그 마이그레이션 (Feature Flag Migration)

피처 플래그를 사용하여 소비자들을 하나씩 이전 시스템에서 새로운 시스템으로 전환합니다:

```typescript
function getTaskService(userId: string): TaskService {
  if (featureFlags.isEnabled('new-task-service', { userId })) {
    return new NewTaskService();
  }
  return new LegacyTaskService();
}
```

## 좀비 코드 (Zombie Code)

좀비 코드는 아무도 소유하지 않지만 모두가 의존하는 코드입니다. 활발히 유지보수되지 않고 명확한 소유 팀이 없으며, 보안 취약점과 호환성 문제가 쌓여갑니다. 징후:

- 6개월 이상 커밋이 없지만 활성 소비자가 존재함
- 할당된 유지보수 담당자나 팀이 없음
- 아무도 고치지 않는 실패하는 테스트들
- 아무도 업데이트하지 않는 알려진 취약점이 있는 의존성들
- 더 이상 존재하지 않는 시스템을 참조하는 문서

**대응:** 담당자를 배정하고 제대로 유지보수하거나, 구체적인 마이그레이션 계획과 함께 폐기하세요. 좀비 코드를 방치해서는 안 됩니다. 투자를 하거나 제거해야 합니다.

## 흔한 자기합리화 (Common Rationalizations)

| 자기합리화 | 실제 상황 |
|---|---|
| "아직 잘 작동하는데 왜 제거하나요?" | 아무도 유지보수하지 않는 코드는 보안 부채와 복잡성을 축적합니다. 유지보수 비용은 조용히 커집니다. |
| "나중에 누군가 필요할 수도 있어요" | 나중에 필요하다면 다시 만들 수 있습니다. 쓰지 않는 코드를 "만약의 경우"를 위해 보관하는 비용이 다시 만드는 비용보다 큽니다. |
| "마이그레이션 비용이 너무 커요" | 마이그레이션 비용과 향후 2~3년간의 지속적인 유지보수 비용을 비교해 보세요. 장기적으로는 마이그레이션이 보통 더 저렴합니다. |
| "새 시스템이 끝나면 폐기할게요" | 폐기 계획은 설계 시점부터 시작됩니다. 새 시스템이 끝날 때쯤이면 새로운 우선순위가 생길 것입니다. 지금 계획하세요. |
| "사용자들이 알아서 마이그레이션하겠죠" | 그렇지 않습니다. 도구, 문서, 인센티브를 제공하거나 직접 마이그레이션을 수행하세요(변경 책임의 원칙). |
| "두 시스템을 영원히 유지할 수 있어요" | 같은 일을 하는 두 시스템은 유지보수, 테스트, 문서화, 온보딩 비용이 두 배로 듭니다. |

## 위험 신호 (Red Flags)

- 대체제 없이 폐기된 시스템
- 마이그레이션 도구나 문서 없이 이루어진 폐기 공지
- 수년째 진전 없이 권고 상태로만 머물러 있는 "소프트" 폐기
- 소유자 없이 활발히 사용되는 좀비 코드
- 폐기된 시스템에 추가되는 새로운 기능 (대신 대체제에 투자하세요)
- 현재 사용량을 측정하지 않은 채 이루어지는 폐기
- 활성 소비자가 0인지 확인하지 않고 코드를 제거함

## 검증 (Verification)

폐기 완료 후:

- [ ] 대체제가 프로덕션 환경에서 입증되었으며 모든 핵심 유즈케이스를 커버하는가
- [ ] 구체적인 단계와 예시가 포함된 마이그레이션 가이드가 존재하는가
- [ ] 모든 활성 소비자가 마이그레이션되었는가 (메트릭/로그로 확인)
- [ ] 이전 코드, 테스트, 문서 및 설정이 완전히 제거되었는가
- [ ] 코드베이스에 폐기된 시스템에 대한 참조가 남아 있지 않는가
- [ ] 폐기 공지가 제거되었는가 (목적을 다했으므로)

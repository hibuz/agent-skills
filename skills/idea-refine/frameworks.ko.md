# Ideation 프레임워크 참조

이 프레임워크들을 선별적으로 사용하세요. 아이디어에 맞는 렌즈를 고르세요 — 모든 프레임워크를 기계적으로 실행하지 마세요. 목표는 체크리스트를 따르는 것이 아니라 사고를 푸는 것입니다.

## SCAMPER

일곱 가지 다른 작업을 적용해 기존 아이디어를 변형하는 구조화된 방법:

- **Substitute(대체):** 어떤 컴포넌트, 재료, 또는 프로세스를 바꿀 수 있나? 핵심 기술을 교체하면? 타겟 청중을? 비즈니스 모델을?
- **Combine(결합):** 이것을 다른 제품, 서비스, 또는 아이디어와 merge하면? 보통 함께 가지 않는 두 가지가 새로운 것을 만든다면?
- **Adapt(적응):** 이것과 비슷한 게 또 뭐가 있나? 다른 산업, 도메인, 또는 시대의 어떤 아이디어를 빌릴 수 있나? 자연에 어떤 평행이 존재하나?
- **Modify(확대/축소):** 10배 크게 하면? 10배 작게? 한 기능을 과장하면? 절대 최소로 strip하면?
- **Put to other uses(다른 용도):** 또 누가 이것을 쓸 수 있나? 어떤 다른 문제를 해결할 수 있나? 완전히 다른 context에서 쓰면 어떻게 되나?
- **Eliminate(제거):** 한 기능을 완전히 제거하면 어떻게 되나? configuration이 0인 버전은? 단계가 절반인 버전은?
- **Reverse/Rearrange(역전/재배열):** 단계를 반대 순서로 하면? 시스템 대신 사용자가 작업하면(또는 반대)? value chain을 역전하면?

**가장 적합한 곳:** 기존 제품/기능 개선이나 재상상. greenfield 아이디어에는 덜 유용.

## How Might We (HMW)

"How Might We..." 형식을 사용해 문제를 기회로 reframe:

- 관찰이나 pain point에서 시작
- "How might we [원하는 결과] for [특정 사용자] without [핵심 제약]?"로 reframe
- 같은 문제의 여러 HMW framing 생성 — 다른 framing이 다른 솔루션을 품

**좋은 HMW 특질:**
- 실행 가능할 만큼 좁음 ("...신규 사용자가 첫 5분에 관련 콘텐츠를 찾게 돕기")
- 창의적 솔루션을 허용할 만큼 넓음 ("...추천 사이드바 추가"가 아니라)
- 창의성을 강제하는 긴장이나 제약을 포함

**나쁜 HMW 특질:**
- 너무 넓음: "사용자를 행복하게 하려면?"
- 너무 좁음: "설정 페이지에 버튼을 추가하려면?"
- 솔루션 내장: "지원용 챗봇을 만들려면?"

**가장 적합한 곳:** 막힌 사고 reframe. 누군가 솔루션에 닻을 내렸을 때, 문제로 끌어당기기.

## First Principles Thinking

아이디어를 근본적 진실로 분해한 뒤, 거기서 재구축:

1. **참이라고 아는 것은?** (가정 아님, 관습 아님 — 실제로 참)
2. **무엇을 가정하나?** 모든 가정을 나열, 명백해 보이는 것도
3. **어떤 가정에 도전할 수 있나?** 각각에 대해: "이게 실제로 물리 법칙인가, 아니면 그냥 해온 방식인가?"
4. **진실에서 재구축.** 근본적 진실만 있다면, 무엇을 만들까?

**가장 적합한 곳:** 점진적 사고에서 벗어나기. 모든 아이디어가 현상의 작은 개선처럼 느껴질 때.

## Jobs to Be Done (JTBD)

사용자가 원한다고 말하는 것이 아니라 달성하려는 것에 집중:

- **기능적 job:** 어떤 task를 완료하려 하나?
- **감정적 job:** 어떻게 느끼고 싶나?
- **사회적 job:** 어떻게 인식되고 싶나?

형식: "When I [상황], I want to [동기], so I can [예상 결과]."

**핵심 통찰:** 사람들은 제품을 사지 않습니다 — job을 하도록 고용합니다. 경쟁 제품이 항상 같은 카테고리에 있는 것은 아닙니다. (Netflix는 다른 스트리밍 서비스만이 아니라 잠과 경쟁.)

**가장 적합한 곳:** 진짜 문제 이해. 올바른 것을 해결하는지 확신 없을 때.

## Constraint-Based Ideation

창의적 솔루션을 강제하기 위해 의도적으로 제약을 부과:

- **시간 제약:** "이걸 빌드하는 데 하루뿐이라면?"
- **기능 제약:** "기능을 하나만 가질 수 있다면?"
- **기술 제약:** "[명백한 기술]을 못 쓴다면?"
- **비용 제약:** "영원히 무료여야 한다면?"
- **청중 제약:** "사용자가 컴퓨터를 한 번도 안 써봤다면?"
- **규모 제약:** "10억 사용자에게 동작해야 한다면? 딱 10명은?"

**가장 적합한 곳:** 복잡도 뚫기. 아이디어가 너무 커지거나 너무 막연해질 때.

## Pre-mortem

아이디어가 이미 실패했다고 상상. 거꾸로 작업:

1. 지금부터 12개월 후. 프로젝트가 ship되고 망함. 무엇이 잘못됐나?
2. 실패의 모든 그럴듯한 이유 나열 — 기술, 시장, 팀, 타이밍
3. 각 실패 모드에: 예방 가능한가? 아이디어가 바뀌어야 한다는 신호인가?
4. 어떤 실패 모드를 받아들일 의향이 있나? 어떤 것이 프로젝트를 죽이나?

**가장 적합한 곳:** Phase 2 평가. 좋게 느껴지지만 압박 테스트 안 된 아이디어 stress-test.

## Analogous Inspiration

다른 도메인이 유사한 문제를 어떻게 해결했는지 보기:

- 어떤 산업이 이 문제의 한 버전을 이미 해결했나?
- [특정 회사/제품]이 만들면 이것은 어떻게 보일까?
- 어떤 자연 시스템이 이렇게 작동하나?
- 어떤 역사적 전례가 존재하나?

핵심은 표면 수준이 아니라 *구조적* 유사성을 찾는 것입니다. "Uber for X"는 표면 수준. "낯선 사람 간 신뢰 문제를 해결하는 양면 마켓플레이스"는 구조적.

**가장 적합한 곳:** Phase 1 확장. 명백한 접근과 진정으로 다르게 느껴지는 변형 생성.

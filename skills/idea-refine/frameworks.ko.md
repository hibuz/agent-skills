# 아이디어 프레임워크 참조 (Ideation Frameworks Reference)

이 프레임워크들을 선택적으로 사용하세요. 모든 프레임워크를 기계적으로 실행하지 말고, 아이디어에 적합한 렌즈를 고르세요. 목표는 체크리스트를 따르는 것이 아니라 사고를 자극하는 것입니다.

## SCAMPER

일곱 가지 다른 작업을 적용하여 기존 아이디어를 변형하는 구조화된 방법입니다.

- **대체 (Substitute):** 어떤 구성 요소, 재료 또는 프로세스를 교체할 수 있나요? 핵심 기술을 바꾼다면 어떨까요? 타겟 오디언스는요? 비즈니스 모델은요?
- **결합 (Combine):** 이것을 다른 제품, 서비스 또는 아이디어와 합치면 어떨까요? 보통 함께 쓰이지 않는 두 가지를 합쳐서 새로운 것을 만들 수 있을까요?
- **응용 (Adapt):** 이것과 비슷한 것이 또 무엇이 있나요? 다른 산업, 분야 또는 시대에서 빌려올 수 있는 아이디어는 무엇인가요? 자연계에는 어떤 평행 우주가 존재하나요?
- **수정/확대/축소 (Modify - Magnify/Minimize):** 10배 더 크게 만들면 어떨까요? 10배 더 작게 만들면요? 한 가지 기능을 과장한다면요? 절대적인 최소치로 깎아낸다면요?
- **용도 변경 (Put to other uses):** 누가 이것을 또 사용할 수 있을까요? 다른 어떤 문제를 해결할 수 있을까요? 완전히 다른 컨텍스트에서 사용한다면 어떤 일이 벌어질까요?
- **제거 (Eliminate):** 한 가지 기능을 완전히 제거하면 어떻게 될까요? 설정이 전혀 없는 버전은 어떨까요? 단계를 절반으로 줄이면 어떤 모습일까요?
- **역전/재배치 (Reverse/Rearrange):** 단계 순서를 반대로 하면 어떨까요? 시스템 대신 사용자가 작업을 수행한다면(또는 그 반대라면) 어떨까요? 가치 사슬(value chain)을 뒤집는다면요?

**적합한 상황:** 기존 제품/기능을 개선하거나 재구상할 때 유용합니다. 완전히 새로운 분야의 아이디어에는 덜 유용할 수 있습니다.

## How Might We (HMW)

문제를 "우리가 어떻게 하면...(How Might We...)" 형식을 사용하여 기회로 재정의합니다.

- 관찰 내용이나 고충점(pain point)에서 시작하세요.
- "[핵심 제약 조건] 없이 [특정 사용자]를 위해 어떻게 하면 [원하는 결과]를 얻을 수 있을까?"로 재정의하세요.
- 동일한 문제에 대해 여러 HMW 프레임워크를 생성하세요. 프레이밍이 다르면 해결책도 달라집니다.

**좋은 HMW의 특성:**
- 실행 가능할 만큼 좁음 ("...신규 사용자가 처음 5분 이내에 관련 콘텐츠를 찾도록 돕기")
- 창의적인 해결책을 허용할 만큼 넓음 ("...추천 사이드바 추가하기"가 아님)
- 창의성을 자극하는 긴장감이나 제약 조건을 포함함

**나쁜 HMW의 특성:**
- 너무 넓음: "우리가 어떻게 하면 사용자를 행복하게 만들 수 있을까?"
- 너무 좁음: "우리가 어떻게 하면 설정 페이지에 버튼을 추가할 수 있을까?"
- 해결책이 내포됨: "우리가 어떻게 하면 지원을 위한 챗봇을 만들 수 있을까?"

**적합한 상황:** 막힌 사고를 전환할 때 유용합니다. 누군가 특정 해결책에 매몰되어 있을 때 문제로 다시 끌어오세요.

## 제1원리 사고 (First Principles Thinking)

아이디어를 근본적인 진실로 분해한 다음, 거기서부터 다시 구축합니다.

1. **우리가 참이라고 알고 있는 것은 무엇인가?** (가정이나 관습이 아닌, 실제로 참인 것)
2. **우리는 무엇을 가정하고 있는가?** 당연하게 느껴지는 것까지 모든 가정을 나열하세요.
3. **어떤 가정에 도전할 수 있는가?** 각 가정에 대해 물으세요. "이것이 실제로 물리 법칙인가, 아니면 그저 지금까지 그렇게 해온 방식인가?"
4. **진실로부터 다시 구축하기.** 근본적인 진실만 가지고 있다면 무엇을 만들겠습니까?

**적합한 상황:** 점진적인 사고(incremental thinking)에서 벗어날 때 유용합니다. 모든 아이디어가 현상 유지에서 조금씩만 개선되는 것처럼 느껴질 때 사용하세요.

## 해결해야 할 과제 (Jobs to Be Done - JTBD)

사용자가 원하는 것이라고 말하는 것이 아니라, 사용자가 성취하려는 것에 집중하세요.

- **기능적 과제 (Functional job):** 어떤 작업을 완료하려고 하나요?
- **감성적 과제 (Emotional job):** 어떤 기분을 느끼고 싶어 하나요?
- **사회적 과제 (Social job):** 다른 사람들에게 어떻게 보이고 싶어 하나요?

형식: "[상황]일 때, 나는 [기대하는 결과]를 위해 [동기]를 원한다."

**핵심 통찰:** 사람들은 제품을 사는 것이 아니라, 과제(job)를 수행하기 위해 제품을 고용(hire)하는 것입니다. 경쟁 제품이 항상 같은 카테고리에 있는 것은 아닙니다. (넷플릭스는 다른 스트리밍 서비스뿐만 아니라 수면과도 경쟁합니다.)

**적합한 상황:** 진짜 문제를 이해할 때 유용합니다. 올바른 문제를 해결하고 있는지 확신이 서지 않을 때 사용하세요.

## 제약 기반 아이디어 발상 (Constraint-Based Ideation)

창의적인 해결책을 강제하기 위해 의도적으로 제약을 부과합니다.

- **시간 제약:** "이것을 만드는 데 하루밖에 없다면?"
- **기능 제약:** "기능이 딱 하나만 있을 수 있다면?"
- **기술 제약:** "뻔한 기술인 [특정 기술]을 사용할 수 없다면?"
- **비용 제약:** "영원히 무료여야 한다면?"
- **대상 제약:** "사용자가 컴퓨터를 한 번도 써본 적이 없다면?"
- **규모 제약:** "10억 명의 사용자를 수용해야 한다면? 반대로 딱 10명뿐이라면?"

**적합한 상황:** 복잡성을 걷어낼 때 유용합니다. 아이디어가 너무 커지거나 모호해질 때 사용하세요.

## 사전 부검 (Pre-mortem)

아이디어가 이미 실패했다고 가정하고 거꾸로 추적합니다.

1. 지금으로부터 12개월 후입니다. 프로젝트가 출시되었지만 완전히 실패했습니다. 무엇이 잘못되었나요?
2. 기술, 시장, 팀, 타이밍 등 실패 가능한 모든 이유를 나열하세요.
3. 각 실패 모드에 대해: 예방 가능한가요? 아이디어를 바꿔야 한다는 신호인가요?
4. 어떤 실패 모드를 수용할 의사가 있나요? 어떤 것이 프로젝트를 중단시킬 요인인가요?

**적합한 상황:** 2단계 평가(Phase 2 evaluation) 시 유용합니다. 좋아 보이지만 압박 테스트를 거치지 않은 아이디어를 검증할 때 사용하세요.

## 유사한 영감 (Analogous Inspiration)

다른 분야에서 유사한 문제를 어떻게 해결했는지 살펴봅니다.

- 어떤 산업에서 이미 이 문제의 변형을 해결했나요?
- [특정 회사/제품]이 이것을 만든다면 어떤 모습일까요?
- 어떤 자연 시스템이 이런 방식으로 작동하나요?
- 어떤 역사적 선례가 있나요?

핵심은 표면적인 유사성이 아니라 **구조적(structural)** 유사성을 찾는 것입니다. "X를 위한 우버"는 표면적입니다. "낯선 사람들 사이의 신뢰 문제를 해결하는 양면 마켓플레이스"가 구조적인 접근입니다.

**적합한 상황:** 1단계 확장(Phase 1 expansion) 시 유용합니다. 뻔한 접근 방식과는 진정으로 다르게 느껴지는 변형을 생성할 때 사용하세요.

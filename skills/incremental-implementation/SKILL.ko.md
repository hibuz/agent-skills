# 점진적 구현 (Incremental Implementation)

## 개요 (Overview)

얇은 수직 슬라이스(thin vertical slices) 형태로 구축하세요 — 한 조각을 구현하고, 테스트하고, 검증한 다음 확장하세요. 한 번에 전체 기능을 구현하려 하지 마세요. 각 증분(increment)은 시스템을 작동 가능하고 테스트 가능한 상태로 유지해야 합니다. 이것이 거대한 기능을 관리 가능하게 만드는 실행 규율입니다.

## 사용 시점

- 다중 파일 변경을 구현할 때
- 작업 분해(task breakdown)를 바탕으로 새로운 기능을 빌드할 때
- 기존 코드를 리팩토링할 때
- 테스트하기 전에 100줄 이상의 코드를 작성하고 싶은 유혹이 들 때마다

**사용하지 않는 경우:** 범위가 이미 최소화된 단일 파일, 단일 함수 변경 시.

## 증분 사이클 (The Increment Cycle)

```
┌──────────────────────────────────────┐
│                                      │
│   구현 ──→ 테스트 ──→ 검증 ──┐  │
│       ▲                           │  │
│       └───── 커밋 ◄─────────────┘  │
│              │                       │
│              ▼                       │
│          다음 슬라이스                 │
│                                      │
└──────────────────────────────────────┘
```

각 슬라이스에 대해:

1. **구현 (Implement):** 기능을 구성하는 가장 작은 완전한 조각을 구현합니다.
2. **테스트 (Test):** 테스트 스위트를 실행합니다 (테스트가 없다면 작성합니다).
3. **검증 (Verify):** 슬라이스가 의도한 대로 작동하는지 확인합니다 (테스트 통과, 빌드 성공, 수동 확인).
4. **커밋 (Commit):** 설명이 포함된 메시지와 함께 진행 상황을 저장합니다 (원자적 커밋에 대한 안내는 `git-workflow-and-versioning.ko.md` 참조).
5. **다음 슬라이스로 이동:** 처음부터 다시 시작하지 않고 계속 이어 나갑니다.

## 슬라이싱 전략 (Slicing Strategies)

### 수직적 슬라이스 (Vertical Slices - 권장됨)

전체 스택을 관통하는 하나의 완전한 경로를 구축하세요:

```
슬라이스 1: 작업 생성 (DB + API + 기본 UI)
    → 테스트 통과, 사용자가 UI를 통해 작업을 생성할 수 있음

슬라이스 2: 작업 목록 표시 (쿼리 + API + UI)
    → 테스트 통과, 사용자가 자신의 작업을 볼 수 있음

슬라이스 3: 작업 편집 (업데이트 + API + UI)
    → 테스트 통과, 사용자가 작업을 수정할 수 있음

슬라이스 4: 작업 삭제 (삭제 + API + UI + 확인창)
    → 테스트 통과, 전체 CRUD 완료
```

각 슬라이스는 작동하는 엔드 투 엔드(end-to-end) 기능을 제공합니다.

### 컨트랙트 우선 슬라이싱 (Contract-First Slicing)

백엔드와 프론트엔드가 병렬로 개발되어야 할 때:

```
슬라이스 0: API 컨트랙트 정의 (타입, 인터페이스, OpenAPI 스펙)
슬라이스 1a: 컨트랙트에 따른 백엔드 구현 + API 테스트
슬라이스 1b: 컨트랙트와 일치하는 모의 데이터(mock data)를 사용한 프론트엔드 구현
슬라이스 2: 통합 및 엔드 투 엔드 테스트
```

### 리스크 우선 슬라이싱 (Risk-First Slicing)

가장 위험하거나 불확실한 부분을 먼저 해결하세요:

```
슬라이스 1: WebSocket 연결 작동 증명 (가장 높은 리스크)
슬라이스 2: 검증된 연결 위에서 실시간 작업 업데이트 기능 구축
슬라이스 3: 오프라인 지원 및 재연결 기능 추가
```

슬라이스 1이 실패하면 슬라이스 2와 3에 투자하기 전에 문제를 발견할 수 있습니다.

## 구현 규칙 (Implementation Rules)

### 규칙 0: 단순함 우선 (Simplicity First)

코드를 작성하기 전에 자문해 보세요: "작동할 수 있는 가장 단순한 것은 무엇인가?"

코드를 작성한 후에는 다음 체크리스트로 검토하세요:
- 더 적은 줄 수로 할 수 있는가?
- 이러한 추상화가 그 복잡성만큼의 가치가 있는가?
- 시니어 엔지니어가 이것을 보고 "왜 그냥... 하지 않았나요?"라고 말할 것인가?
- 가상의 미래 요구 사항이 아니라 현재 작업에 맞게 구축하고 있는가?

```
단순성 체크:
✗ 하나의 알림을 위해 미들웨어 파이프라인이 있는 제네릭 EventBus 사용
✓ 단순한 함수 호출

✗ 두 개의 유사한 컴포넌트를 위한 추상 팩토리 패턴 사용
✓ 공유 유틸리티를 가진 두 개의 직관적인 컴포넌트

✗ 세 개의 폼을 위해 설정 기반의 폼 빌더 사용
✓ 세 개의 폼 컴포넌트 작성
```

유사한 세 줄의 코드가 성급한 추상화보다 낫습니다. 나이브(naive)하고 명백하게 올바른 버전을 먼저 구현하세요. 테스트로 정확성이 증명된 후에만 최적화하세요.

### 규칙 0.5: 범위 규율 (Scope Discipline)

작업이 요구하는 부분만 수정하세요.

다음을 수행하지 마세요:
- 변경 사항 옆에 있는 코드 "정리"
- 수정하지 않는 파일의 임포트(imports) 리팩토링
- 완전히 이해하지 못하는 주석 제거
- "유용해 보여서" 스펙에 없는 기능 추가
- 읽기만 하는 파일의 문법 최신화

작업 범위 밖에서 개선할 가치가 있는 것을 발견하면 수정하지 말고 메모하세요:

```
발견했지만 수정하지 않음:
- src/utils/format.ts에 사용되지 않는 임포트가 있음 (이 작업과 무관)
- 인증 미들웨어의 에러 메시지를 개선할 수 있음 (별도 작업)
→ 이들을 위한 작업을 생성할까요?
```

### 규칙 1: 한 번에 한 가지만 (One Thing at a Time)

각 증분은 하나의 논리적인 것만 변경합니다. 관심사를 섞지 마세요:

**나쁨:** 하나의 커밋에 새로운 컴포넌트 추가, 기존 컴포넌트 리팩토링, 빌드 설정 업데이트를 모두 포함.

**좋음:** 각 변경 사항에 대해 하나씩, 세 개의 별도 커밋으로 분리.

### 규칙 2: 컴파일 가능한 상태 유지 (Keep It Compilable)

각 증분 후에 프로젝트가 빌드되어야 하며 기존 테스트가 통과되어야 합니다. 슬라이스 사이에 코드베이스를 망가진 상태로 두지 마세요.

### 규칙 3: 미완성 기능을 위한 피처 플래그 (Feature Flags)

기능이 사용자에게 공개될 준비가 되지 않았지만 증분을 머지해야 하는 경우:

```typescript
// 작업 중인 기능을 위한 피처 플래그
const ENABLE_TASK_SHARING = process.env.FEATURE_TASK_SHARING === 'true';

if (ENABLE_TASK_SHARING) {
  // 새로운 공유 UI
}
```

이를 통해 미완성된 작업을 노출하지 않고 메인 브랜치에 작은 증분들을 머지할 수 있습니다.

### 규칙 4: 안전한 기본값 (Safe Defaults)

새로운 코드는 기본적으로 안전하고 보수적인 동작을 수행해야 합니다:

```typescript
// 안전함: 기본적으로 비활성화, 선택적 사용(opt-in)
export function createTask(data: TaskInput, options?: { notify?: boolean }) {
  const shouldNotify = options?.notify ?? false;
  // ...
}
```

### 규칙 5: 롤back 친화적 (Rollback-Friendly)

각 증분은 독립적으로 되돌릴 수 있어야 합니다:

- 추가적인 변경(새 파일, 새 함수)은 되돌리기 쉽습니다.
- 기존 코드에 대한 수정은 최소화하고 집중되어야 합니다.
- 데이터베이스 마이그레이션에는 대응하는 롤백 마이그레이션이 있어야 합니다.
- 한 커밋에서 무언가를 삭제하고 동시에 교체하지 마세요 — 이들을 분리하세요.

## 에이전트와 함께 작업하기

에이전트에게 점진적으로 구현하도록 지시할 때:

```
"계획의 작업 3을 구현해 봅시다.

먼저 데이터베이스 스키마 변경과 API 엔드포인트만 시작하세요.
아직 UI는 건드리지 마세요 — 다음 증분에서 할 것입니다.

구현 후에는 `npm test`와 `npm run build`를 실행하여
망가진 것이 없는지 확인하세요."
```

각 증분에서 무엇이 범위 내에 있고 무엇이 범위 밖인지 명시적으로 지시하세요.

## 증분 체크리스트 (Increment Checklist)

각 증분 후에 다음을 확인하세요:

- [ ] 변경 사항이 한 가지 작업만 수행하며 완전히 완료됨
- [ ] 모든 기존 테스트가 여전히 통과함 (`npm test`)
- [ ] 빌드가 성공함 (`npm run build`)
- [ ] 타입 체크가 통과함 (`npx tsc --noEmit`)
- [ ] 린트가 통과함 (`npm run lint`)
- [ ] 새로운 기능이 예상대로 작동함
- [ ] 변경 사항이 서술적인 메시지와 함께 커밋됨

## 일반적인 합리화

| 합리화 | 실제 |
|---|---|
| "코드가 다 짜여진 후에 테스트할게요" | 버그는 누적됩니다. 슬라이스 1의 버그는 슬라이스 2-5를 잘못되게 만듭니다. 각 슬라이스를 테스트하세요. |
| "한꺼번에 다 하는 게 더 빨라요" | 무언가 고장 나기 전까지는 그렇게 *느껴질* 뿐입니다. 고장이 나면 변경된 500줄 중 무엇이 원인인지 찾을 수 없게 됩니다. |
| "이 변경사항들은 너무 작아서 따로 커밋하기 민망해요" | 작은 커밋은 비용이 들지 않습니다. 큰 커밋은 버그를 숨기고 롤백을 고통스럽게 만듭니다. |
| "피처 플래그는 나중에 추가할게요" | 기능이 완성되지 않았다면 사용자에게 보여서는 안 됩니다. 지금 플래그를 추가하세요. |
| "이 리팩토링은 같이 포함해도 될 만큼 작아요" | 기능과 섞인 리팩토링은 리뷰와 디버깅을 어렵게 만듭니다. 이들을 분리하세요. |

## 레드 플래그 (Red Flags)

- 테스트를 실행하지 않고 100줄 이상의 코드를 작성함
- 하나의 증분에 여러 개의 무관한 변경 사항이 포함됨
- "이것도 그냥 빨리 추가할게요" 식의 범위 확장
- 더 빨리 진행하기 위해 테스트/검증 단계를 건너뜀
- 증분 사이에 빌드나 테스트가 깨진 상태로 방치됨
- 커밋되지 않은 거대한 변경 사항이 쌓임
- 세 번째 유스케이스가 생기기도 전에 추상화를 구축함
- "온 김에" 작업 범위 밖의 파일을 수정함
- 일회성 작업을 위해 새로운 유틸리티 파일을 생성함

## 검증 (Verification)

한 작업에 대한 모든 증분을 완료한 후:

- [ ] 각 증분이 개별적으로 테스트되고 커밋됨
- [ ] 전체 테스트 스위트가 통과함
- [ ] 빌드가 클린함
- [ ] 기능이 명세대로 처음부터 끝까지 작동함
- [ ] 커밋되지 않은 변경 사항이 남아있지 않음

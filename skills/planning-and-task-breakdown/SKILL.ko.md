# 계획 및 작업 분해 (Planning and Task Breakdown)

## 개요 (Overview)

작업을 명시적인 수락 기준(acceptance criteria)을 가진 작고 검증 가능한 단위로 분해하세요. 좋은 작업 분해는 작업을 안정적으로 완료하는 에이전트와 뒤얽힌 엉망진창을 만드는 에이전트를 결정짓는 차이점입니다. 모든 작업은 하나의 집중된 세션에서 구현, 테스트 및 검증할 수 있을 만큼 작아야 합니다.

## 사용 시점

- 스펙이 있고 이를 구현 가능한 단위로 나누어야 할 때
- 작업이 시작하기에 너무 크거나 모호하게 느껴질 때
- 여러 에이전트나 세션에 걸쳐 작업을 병렬화해야 할 때
- 인간에게 작업 범위를 전달해야 할 때
- 구현 순서가 명확하지 않을 때

**사용하지 않는 경우:** 범위가 명확한 단일 파일 변경, 또는 스펙에 이미 잘 정의된 작업들이 포함되어 있는 경우.

## 계획 프로세스

### 1단계: 계획 모드 진입 (Enter Plan Mode)

코드를 작성하기 전에 읽기 전용 모드로 작업하세요:

- 스펙과 관련 코드 섹션을 읽습니다.
- 기존 패턴과 컨벤션(conventions)을 식별합니다.
- 컴포넌트 간의 의존성을 매핑합니다.
- 리스크와 미지의 요소(unknowns)를 메모합니다.

**계획 단계에서는 코드를 작성하지 마세요.** 출력물은 구현이 아니라 계획 문서여야 합니다.

### 2단계: 의존성 그래프 식별 (Identify the Dependency Graph)

무엇이 무엇에 의존하는지 매핑하세요:

```
데이터베이스 스키마
    │
    ├── API 모델/타입
    │       │
    │       ├── API 엔드포인트
    │       │       │
    │       │       └── 프론트엔드 API 클라이언트
    │       │               │
    │       │               └── UI 컴포넌트
    │       │
    │       └── 검증 로직
    │
    └── 시드 데이터 / 마이그레이션
```

구현 순서는 의존성 그래프를 아래에서 위로 따라갑니다: 기초(foundations)를 먼저 구축하세요.

### 3단계: 수직적 슬라이싱 (Slice Vertically)

데이터베이스 전체를 만들고, 그다지 API 전체를 만들고, 마지막에 UI 전체를 만드는 대신 — 한 번에 하나의 완전한 기능 경로(feature path)를 구축하세요:

**나쁜 예 (수평적 슬라이싱):**
```
작업 1: 전체 데이터베이스 스키마 구축
작업 2: 모든 API 엔드포인트 구축
작업 3: 모든 UI 컴포넌트 구축
작업 4: 모든 기능 연결
```

**좋은 예 (수직적 슬라이싱):**
```
작업 1: 사용자 계정 생성 기능 (회원가입을 위한 스키마 + API + UI)
작업 2: 사용자 로그인 기능 (인증 스키마 + API + UI)
작업 3: 작업 생성 기능 (작업 스키마 + API + UI)
작업 4: 작업 목록 보기 기능 (쿼리 + API + UI)
```

각 수직적 슬라이스는 작동하고 테스트 가능한 기능을 제공합니다.

### 4단계: 작업 작성 (Write Tasks)

각 작업은 다음 구조를 따릅니다:

```markdown
## 작업 [N]: [짧고 서술적인 제목]

**설명:** 이 작업이 무엇을 달성하는지 설명하는 한 단락.

**수락 기준:**
- [ ] [구체적이고 테스트 가능한 조건]
- [ ] [구체적이고 테스트 가능한 조건]

**검증 방법:**
- [ ] 테스트 통과: `npm test -- --grep "feature-name"`
- [ ] 빌드 성공: `npm run build`
- [ ] 수동 확인: [검증할 내용에 대한 설명]

**의존성:** [이 작업이 의존하는 작업 번호, 또는 "없음"]

**수정될 가능성이 높은 파일:**
- `src/path/to/file.ts`
- `tests/path/to/test.ts`

**예상 범위:** [Small: 1-2개 파일 | Medium: 3-5개 파일 | Large: 5개 이상 파일]
```

### 5단계: 순서 지정 및 체크포인트 (Order and Checkpoint)

다음과 같이 작업을 배치하세요:

1. 의존성이 충족되도록 함 (기초를 먼저 구축)
2. 각 작업이 완료된 후 시스템이 작동 가능한 상태를 유지하도록 함
3. 매 2-3개의 작업마다 검증 체크포인트를 둠
4. 리스크가 높은 작업을 앞쪽에 배치 (빨리 실패하기)

명시적인 체크포인트를 추가하세요:

```markdown
## 체크포인트: 작업 1-3 완료 후
- [ ] 모든 테스트 통과
- [ ] 에러 없이 애플리케이션 빌드됨
- [ ] 핵심 사용자 흐름이 처음부터 끝까지 작동함
- [ ] 계속 진행하기 전 인간 엔지니어와 리뷰
```

## 작업 크기 가이드라인

| 크기 | 파일 수 | 범위 | 예시 |
|------|-------|-------|---------|
| **XS** | 1 | 단일 함수 또는 설정 변경 | 검증 규칙 추가 |
| **S** | 1-2 | 하나의 컴포넌트 또는 엔드포인트 | 새로운 API 엔드포인트 추가 |
| **M** | 3-5 | 하나의 기능 슬라이스 | 사용자 등록 흐름 |
| **L** | 5-8 | 다중 컴포넌트 기능 | 필터링과 페이지네이션이 포함된 검색 |
| **XL** | 8+ | **너무 큼 — 더 세분화하세요** | — |

작업이 L 이상인 경우 더 작은 작업으로 나누어야 합니다. 에이전트는 S와 M 크기의 작업에서 가장 좋은 성능을 발휘합니다.

**작업을 더 세분화해야 하는 시점:**
- 하나의 집중된 세션(에이전트 작업 기준 약 2시간 이상)보다 오래 걸릴 것 같을 때
- 수락 기준을 3개 이하의 불렛 포인트로 설명할 수 없을 때
- 두 개 이상의 독립적인 하위 시스템(예: 인증과 결제)을 건드릴 때
- 작업 제목에 "그리고(and)"를 쓰고 있는 자신을 발견할 때 (두 개의 작업이라는 신호)

## 계획 문서 템플릿

```markdown
# 구현 계획: [기능/프로젝트 이름]

## 개요
[우리가 구축하려는 내용에 대한 한 단락 요약]

## 아키텍처 결정 사항
- [주요 결정 1 및 근거]
- [주요 결정 2 및 근거]

## 작업 목록

### 1단계: 기초 (Foundation)
- [ ] 작업 1: ...
- [ ] 작업 2: ...

### 체크포인트: 기초
- [ ] 테스트 통과, 빌드 클린

### 2단계: 핵심 기능 (Core Features)
- [ ] 작업 3: ...
- [ ] 작업 4: ...

### 체크포인트: 핵심 기능
- [ ] 엔드 투 엔드 흐름 작동 확인

### 3단계: 마무리 (Polish)
- [ ] 작업 5: ...
- [ ] 작업 6: ...

### 체크포인트: 완료
- [ ] 모든 수락 기준 충족
- [ ] 리뷰 준비 완료

## 리스크 및 완화 전략
| 리스크 | 영향도 | 완화 전략 |
|------|--------|------------|
| [리스크 내용] | [High/Med/Low] | [전략] |

## 열린 질문
- [인간의 입력이 필요한 질문]
```

## 병렬화 기회

여러 에이전트나 세션을 사용할 수 있는 경우:

- **병렬화 가능:** 독립적인 기능 슬라이스, 이미 구현된 기능에 대한 테스트, 문서화
- **순차적 진행 필수:** 데이터베이스 마이그레이션, 공유 상태 변경, 의존성 체인
- **조정 필요:** API 컨트랙트를 공유하는 기능들 (먼저 컨트랙트를 정의한 후 병렬화)

## 일반적인 합리화

| 합리화 | 실제 |
|---|---|
| "진행하면서 생각할게요" | 이는 뒤얽힌 엉망진창과 재작업으로 끝나는 지름길입니다. 10분의 계획이 몇 시간을 아껴줍니다. |
| "작업들이 뻔해요" | 그래도 적으세요. 명시적인 작업은 숨겨진 의존성과 잊어버린 엣지 케이스를 드러냅니다. |
| "계획은 오버헤드예요" | 계획이 곧 작업입니다. 계획 없는 구현은 그냥 타이핑일 뿐입니다. |
| "머릿속에 다 들어있어요" | 컨텍스트 윈도우(context windows)는 유한합니다. 기록된 계획은 세션 경계를 넘어 유지됩니다. |
| "스펙에 다 적혀 있어요" | 스펙은 *무엇을* 할지 말해주고, 계획은 그것을 *어떻게* 실행할지 말해줍니다. |

## 레드 플래그 (Red Flags)

- 서면 작업 목록 없이 구현을 시작함
- 수락 기준 없이 "기능 구현"이라고만 적힌 작업
- 계획에 검증 단계가 없음
- 모든 작업이 XL 크기임
- 작업 사이에 체크포인트가 없음
- 의존성 순서가 고려되지 않음

## 검증 (Verification)

구현을 시작하기 전에 다음을 확인하세요:

- [ ] 모든 작업에 수락 기준이 있음
- [ ] 모든 작업에 검증 단계가 있음
- [ ] 작업 의존성이 식별되고 올바르게 순서가 정해짐
- [ ] 어떤 작업도 약 5개 이상의 파일을 수정하지 않음
- [ ] 주요 단계 사이에 체크포인트가 존재함
- [ ] 인간 엔지니어가 계획을 리뷰하고 승인함

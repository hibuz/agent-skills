# 출시 및 배포 (Shipping and Launch)

## 개요 (Overview)

자신 있게 출시하세요. 목표는 단순히 배포하는 것이 아니라 모니터링 체계를 갖추고, 롤백 계획을 준비하며, 성공의 모습이 무엇인지 명확히 이해한 상태에서 안전하게 배포하는 것입니다. 모든 출시(launch)는 가역적이고, 관찰 가능하며, 점진적이어야 합니다.

## 사용 시점

- 기능을 처음으로 프로덕션에 배포할 때
- 사용자에게 중대한 변경 사항을 릴리스할 때
- 데이터나 인프라를 마이그레이션할 때
- 베타 또는 얼리 액세스 프로그램을 시작할 때
- 리스크가 있는 모든 배포 시 (사실 모든 배포가 리스크가 있습니다)

## 출시 전 체크리스트 (The Pre-Launch Checklist)

### 코드 품질 (Code Quality)

- [ ] 모든 테스트 통과 (단위, 통합, e2e)
- [ ] 경고 없이 빌드 성공
- [ ] 린트 및 타입 체크 통과
- [ ] 코드 리뷰 및 승인 완료
- [ ] 출시 전 해결되어야 할 TODO 주석 없음
- [ ] 프로덕션 코드에 `console.log` 디버깅 문 없음
- [ ] 예상되는 실패 모드를 다루는 에러 처리 완료

### 보안 (Security)

- [ ] 코드나 버전 관리 시스템에 비밀 정보(secrets) 없음
- [ ] `npm audit` 결과 치명적(critical) 또는 높음(high) 취약점 없음
- [ ] 모든 사용자 대면 엔드포인트에 입력 검증 적용
- [ ] 인증 및 인가 체크가 적절히 이루어짐
- [ ] 보안 헤더 설정 완료 (CSP, HSTS 등)
- [ ] 인증 엔드포인트에 속도 제한(rate limiting) 적용
- [ ] CORS가 특정 오리진으로 설정됨 (와일드카드 `*` 아님)

### 성능 (Performance)

- [ ] Core Web Vitals가 "Good" 임계값 내에 있음
- [ ] 핵심 경로에 N+1 쿼리 없음
- [ ] 이미지 최적화 완료 (압축, 반응형 크기, 지연 로딩)
- [ ] 번들 크기가 예산 내에 있음
- [ ] 데이터베이스 쿼리에 적절한 인덱스 생성됨
- [ ] 정적 자산 및 반복 쿼리에 캐싱 설정 완료

### 접근성 (Accessibility)

- [ ] 모든 대화형 엘리먼트에서 키보드 탐색 작동
- [ ] 스크린 리더가 페이지 내용과 구조를 전달할 수 있음
- [ ] 색상 대비가 WCAG 2.1 AA (텍스트 4.5:1)를 충족함
- [ ] 모달 및 동적 컨텐츠의 포커스 관리 적절함
- [ ] 에러 메시지가 서술적이며 폼 필드와 연관됨
- [ ] axe-core 또는 Lighthouse에서 접근성 경고 없음

### 인프라 (Infrastructure)

- [ ] 프로덕션 환경 변수 설정 완료
- [ ] 데이터베이스 마이그레이션 적용됨 (또는 적용 준비 완료)
- [ ] DNS 및 SSL 설정 완료
- [ ] 정적 자산을 위한 CDN 설정 완료
- [ ] 로깅 및 에러 리포팅 설정 완료
- [ ] 헬스 체크(health check) 엔드포인트 존재 및 응답 확인

### 문서화 (Documentation)

- [ ] 새로운 설정 요구 사항으로 README 업데이트됨
- [ ] API 문서 최신화 완료
- [ ] 아키텍처 결정 사항에 대해 ADR 작성됨
- [ ] 변경 로그(Changelog) 업데이트됨
- [ ] 사용자용 문서 업데이트됨 (해당하는 경우)

## 피처 플래그 전략 (Feature Flag Strategy)

배포와 릴리스를 분리하기 위해 피처 플래그 뒤에서 배포하세요:

```typescript
// 피처 플래그 확인
const flags = await getFeatureFlags(userId);

if (flags.taskSharing) {
  // 새로운 기능: 작업 공유
  return <TaskSharingPanel task={task} />;
}

// 기본값: 기존 동작
return null;
```

**피처 플래그 생명주기:**

```
1. 플래그 OFF 상태로 배포       → 코드는 프로덕션에 있지만 비활성 상태
2. 팀/베타 유저에게 활성화      → 프로덕션 환경에서 내부 테스트
3. 단계적 출시 (GRADUAL ROLLOUT) → 5% → 25% → 50% → 100% 사용자에게 적용
4. 각 단계별 모니터링          → 에러율, 성능, 사용자 피드백 관찰
5. 정리 (CLEAN UP)            → 완전 출시 후 플래그 및 데드 코드 경로 제거
```

**규칙:**
- 모든 피처 플래그에는 소유자와 만료일이 있어야 합니다.
- 완전 출시 후 2주 이내에 플래그를 정리하세요.
- 피처 플래그를 중첩하지 마세요 (조합의 수가 기하급수적으로 늘어납니다).
- CI에서 두 가지 플래그 상태(ON 및 OFF)를 모두 테스트하세요.

## 단계적 출시 (Staged Rollout)

### 출시 순서 (The Rollout Sequence)

```
1. 스테이징(Staging)에 배포
   └── 스테이징 환경에서 전체 테스트 스위트 실행
   └── 핵심 흐름에 대한 수동 스모크 테스트(smoke test)

2. 프로덕션에 배포 (피처 플래그 OFF)
   └── 배포 성공 확인 (헬스 체크)
   └── 에러 모니터링 확인 (새로운 에러 없음)

3. 팀에게 활성화 (내부 사용자에게 플래그 ON)
   └── 팀원이 프로덕션에서 기능을 사용
   └── 24시간 모니터링 기간

4. 카나리(CANARY) 출시 (사용자의 5%에게 플래그 ON)
   └── 에러율, 지연 시간, 사용자 동작 모니터링
   └── 지표 비교: 카나리군 vs 대조군
   └── 24-48시간 모니터링 기간
   └── 모든 임계값을 통과한 경우에만 진행 (아래 테이블 참조)

5. 점진적 확대 (25% -> 50% -> 100%)
   └── 각 단계마다 동일한 모니터링 수행
   └── 언제든 이전 비율로 롤백할 수 있는 능력 유지

6. 완전 출시 (모든 사용자에게 플래그 ON)
   └── 1주일간 모니터링
   └── 피처 플래그 정리
```

### 출시 결정 임계값 (Rollout Decision Thresholds)

각 단계에서 진행, 유지 또는 롤백 여부를 결정하기 위해 다음 임계값을 사용하세요:

| 지표 | 진행 (Green) | 유지 및 조사 (Yellow) | 롤백 (Red) |
|--------|-----------------|-------------------------------|-----------------|
| 에러율 | 대조군의 10% 이내 | 대조군보다 10-100% 높음 | 대조군의 2배 초과 |
| P95 지연 시간 | 대조군의 20% 이내 | 대조군보다 20-50% 높음 | 대조군의 50% 초과 |
| 클라이언트 JS 에러 | 새로운 에러 유형 없음 | 세션의 0.1% 미만에서 발생 | 세션의 0.1% 이상에서 발생 |
| 비즈니스 지표 | 중립 또는 긍정적 | 5% 미만 하락 (노이즈일 수 있음) | 5% 초과 하락 |

### 롤백 시점

다음의 경우 즉시 롤백하세요:
- 에러율이 대조군의 2배를 초과할 때
- P95 지연 시간이 50% 이상 증가할 때
- 사용자가 보고한 이슈가 급증할 때
- 데이터 무결성 이슈가 감지될 때
- 보안 취약점이 발견될 때

## 모니터링 및 관측 가능성 (Monitoring and Observability)

### 모니터링 대상

```
애플리케이션 지표:
├── 에러율 (전체 및 엔드포인트별)
├── 응답 시간 (p50, p95, p99)
├── 요청 볼륨
├── 활성 사용자 수
└── 주요 비즈니스 지표 (전환율, 참여도)

인프라 지표:
├── CPU 및 메모리 사용률
├── 데이터베이스 커넥션 풀 사용량
├── 디스크 공간
├── 네트워크 지연 시간
└── 큐 깊이 (해당하는 경우)

클라이언트 지표:
├── Core Web Vitals (LCP, INP, CLS)
├── 자바스크립트 에러
├── 클라이언트 관점의 API 에러율
└── 페이지 로드 시간
```

### 에러 리포팅

```typescript
// 리포팅 기능이 포함된 에러 바운더리 설정
class ErrorBoundary extends React.Component {
  componentDidCatch(error: Error, info: React.ErrorInfo) {
    // 에러 트래킹 서비스에 보고
    reportError(error, {
      componentStack: info.componentStack,
      userId: getCurrentUser()?.id,
      page: window.location.pathname,
    });
  }

  render() {
    if (this.state.hasError) {
      return <ErrorFallback onRetry={() => this.setState({ hasError: false })} />;
    }
    return this.props.children;
  }
}

// 서버 측 에러 리포팅
app.use((err: Error, req: Request, res: Response, next: NextFunction) => {
  reportError(err, {
    method: req.method,
    url: req.url,
    userId: req.user?.id,
  });

  // 사용자에게 내부 정보를 노출하지 않음
  res.status(500).json({
    error: { code: 'INTERNAL_ERROR', message: 'Something went wrong' },
  });
});
```

### 출시 후 검증

출시 후 첫 1시간 동안:

```
1. 헬스 엔드포인트가 200을 반환하는지 확인
2. 에러 모니터링 대시보드 확인 (새로운 에러 유형 없음)
3. 지연 시간 대시보드 확인 (성능 저하 없음)
4. 핵심 사용자 흐름을 수동으로 테스트
5. 로그가 정상적으로 쌓이고 읽히는지 확인
6. 롤백 매커니즘 작동 확인 (가능하다면 드라이 런 수행)
```

## 롤백 전략 (Rollback Strategy)

모든 배포는 실행 전에 롤백 계획이 있어야 합니다:

```markdown
## [기능/릴리스] 롤백 계획

### 트리거 조건
- 에러율 > 대조군의 2배
- P95 지연 시간 > [X]ms
- [특정 이슈]에 대한 사용자 보고

### 롤백 단계
1. 피처 플래그 비활성화 (해당하는 경우)
   또는
1. 이전 버전 배포: `git revert <commit> && git push`
2. 롤백 확인: 헬스 체크, 에러 모니터링
3. 커뮤니케이션: 팀에 롤백 사실 공지

### 데이터베이스 고려 사항
- 마이그레이션 [X]의 롤백 명령어: `npx prisma migrate rollback`
- 새로운 기능으로 삽입된 데이터: [보존 / 정리]

### 롤백 소요 시간
- 피처 플래그: < 1분
- 이전 버전 재배포: < 5분
- 데이터베이스 롤백: < 15분
```

## 일반적인 합리화

| 합리화 | 실제 |
|---|---|
| "스테이징에서 작동했으니 프로덕션에서도 작동할 거예요" | 프로덕션은 데이터, 트래픽 패턴, 엣지 케이스가 다릅니다. 배포 후에도 모니터링하세요. |
| "이 기능에는 피처 플래그가 필요 없어요" | 모든 기능에는 킬 스위치(kill switch)가 있는 것이 좋습니다. "단순한" 변경도 문제를 일으킬 수 있습니다. |
| "모니터링은 오버헤드예요" | 모니터링이 없다는 것은 대시보드가 아니라 사용자 불만을 통해 문제를 발견하게 된다는 뜻입니다. |
| "모니터링은 나중에 추가할게요" | 출시 전에 추가하세요. 볼 수 없는 것은 디버깅할 수 없습니다. |
| "롤백하는 것은 실패를 인정하는 거예요" | 롤백은 책임감 있는 엔지니어링입니다. 고장난 기능을 그대로 두는 것이 실패입니다. |

## 레드 플래그 (Red Flags)

- 롤백 계획 없이 배포함
- 프로덕션에 모니터링이나 에러 리포팅이 없음
- 빅뱅 릴리스 (스테이징 없이 한 번에 모든 것을 배포)
- 만료일이나 소유자가 없는 피처 플래그
- 첫 1시간 동안 아무도 배포를 모니터링하지 않음
- 프로덕션 환경 설정을 코드가 아닌 기억에 의존하여 수행함
- "금요일 오후네요, 그냥 배포합시다"

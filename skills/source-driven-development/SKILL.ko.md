# 소스 주도 개발 (Source-Driven Development)

## 개요 (Overview)

프레임워크와 관련된 모든 코드 결정은 공식 문서(official documentation)에 의해 뒷받침되어야 합니다. 기억에 의존하여 구현하지 마세요 — 검증하고, 인용하고, 사용자가 당신의 출처를 볼 수 있게 하세요. 학습 데이터는 오래되고, API는 지원 중단(deprecated)되며, 베스트 프랙티스는 진화합니다. 이 스킬은 모든 패턴이 사용자가 직접 확인할 수 있는 권위 있는 출처를 따르기 때문에 사용자가 신뢰할 수 있는 코드를 받을 수 있도록 보장합니다.

## 사용 시점

- 사용자가 특정 프레임워크의 현재 베스트 프랙티스를 따르는 코드를 원할 때
- 프로젝트 전반에 걸쳐 복사될 보일러플레이트, 스타터 코드 또는 패턴을 구축할 때
- 사용자가 명시적으로 문서화되고 검증된, 혹은 "정확한" 구현을 요청할 때
- 프레임워크가 권장하는 접근 방식이 중요한 기능을 구현할 때 (폼, 라우팅, 데이터 페칭, 상태 관리, 인증 등)
- 프레임워크 특정 패턴을 사용하는 코드를 리뷰하거나 개선할 때
- 기억에 의존하여 프레임워크 특정 코드를 작성하려고 할 때마다

**사용하지 않는 경우:**

- 정확성이 특정 버전에 의존하지 않을 때 (변수 이름 변경, 오타 수정, 파일 이동)
- 모든 버전에서 동일하게 작동하는 순수 로직 (루프, 조건문, 데이터 구조)
- 사용자가 검증보다 속도를 명시적으로 원할 때 ("그냥 빨리 해주세요")

## 프로세스

```
탐지 (DETECT) ──→ 가져오기 (FETCH) ──→ 구현 (IMPLEMENT) ──→ 인용 (CITE)
  │               │                │                 │
  ▼               ▼                ▼                 ▼
 어떤 스택인가?    관련 문서를       문서화된           출처 표시
                 가져옴            패턴 준수
```

### 1단계: 스택 및 버전 탐지 (Detect Stack and Versions)

정확한 버전을 식별하기 위해 프로젝트의 의존성 파일을 읽으세요:

```
package.json    → Node/React/Vue/Angular/Svelte
composer.json   → PHP/Symfony/Laravel
requirements.txt / pyproject.toml → Python/Django/Flask
go.mod          → Go
Cargo.toml      → Rust
Gemfile         → Ruby/Rails
```

찾은 내용을 명시적으로 밝히세요:

```
탐지된 스택:
- React 19.1.0 (package.json 기준)
- Vite 6.2.0
- Tailwind CSS 4.0.3
→ 관련 패턴에 대한 공식 문서를 가져옵니다.
```

버전이 누락되었거나 모호한 경우 **사용자에게 물어보세요**. 추측하지 마세요 — 버전에 따라 올바른 패턴이 결정됩니다.

### 2단계: 공식 문서 가져오기 (Fetch Official Documentation)

구현하려는 기능에 대한 특정 문서 페이지를 가져오세요. 홈페이지나 전체 문서가 아니라 *관련된* 페이지여야 합니다.

**출처 계층 구조 (권위 있는 순서대로):**

| 우선순위 | 출처 | 예시 |
|----------|--------|---------|
| 1 | 공식 문서 (Official documentation) | react.dev, docs.djangoproject.com, symfony.com/doc |
| 2 | 공식 블로그 / 변경 로그 | react.dev/blog, nextjs.org/blog |
| 3 | 웹 표준 참조 자료 | MDN, web.dev, html.spec.whatwg.org |
| 4 | 브라우저/런타임 호환성 | caniuse.com, node.green |

**권위 없음 — 기본 소스로 절대 인용하지 마세요:**

- Stack Overflow 답변
- 블로그 포스트 또는 튜토리얼 (유명한 것이라도 제외)
- AI가 생성한 문서 또는 요약
- 당신 자신의 학습 데이터 (이것을 검증하는 것이 이 스킬의 목적입니다)

**가져올 대상을 정확히 하세요:**

```
나쁜 예: React 홈페이지를 가져옴
좋은 예: react.dev/reference/react/useActionState 페이지를 가져옴

나쁜 예: "django authentication best practices" 검색
좋은 예: docs.djangoproject.com/en/6.0/topics/auth/ 페이지를 가져옴
```

가져온 후에는 핵심 패턴을 추출하고 지원 중단 경고나 마이그레이션 가이드가 있는지 확인하세요.

공식 출처 간에 충돌이 있는 경우(예: 마이그레이션 가이드가 API 참조 문서와 모순되는 경우), 해당 불일치를 사용자에게 알리고 탐지된 버전에서 실제로 작동하는 패턴이 무엇인지 확인하세요.

### 3단계: 문서화된 패턴에 따라 구현 (Implement Following Documented Patterns)

문서에 표시된 내용과 일치하도록 코드를 작성하세요:

- 기억이 아니라 문서의 API 시그니처를 사용하세요.
- 문서에서 새로운 방식을 보여준다면 그 방식을 사용하세요.
- 문서에서 특정 패턴을 지원 중단했다면 해당 버전을 사용하지 마세요.
- 문서에서 다루지 않는 내용이 있다면 미검증 상태임을 표시하세요.

**문서가 기존 프로젝트 코드와 충돌할 때:**

```
충돌 탐지:
기존 코드베이스는 폼 로딩 상태를 위해 useState를 사용하고 있지만,
React 19 문서는 이 패턴에 useActionState를 권장합니다.
(출처: react.dev/reference/react/useActionState)

선택지:
A) 최신 패턴(useActionState) 사용 — 현재 문서와 일치
B) 기존 코드(useState)와 일치시킴 — 코드베이스와 일치
→ 어떤 접근 방식을 선호하시나요?
```

충돌을 드러내세요. 조용히 하나를 선택하지 마세요.

### 4단계: 출처 인용 (Cite Your Sources)

프레임워크 특정 패턴마다 인용을 추가하세요. 사용자가 모든 결정을 검증할 수 있어야 합니다.

**코드 주석에서:**

```typescript
// useActionState를 사용한 React 19 폼 처리
// 출처: https://react.dev/reference/react/useActionState#usage
const [state, formAction, isPending] = useActionState(submitOrder, initialState);
```

**대화에서:**

```
폼 제출 상태를 위해 수동으로 useState를 사용하는 대신 useActionState를 사용합니다.
React 19에서는 수동 isPending/setIsPending 패턴이 이 훅으로 대체되었습니다.

출처: https://react.dev/blog/2024/12/05/react-19#actions
"useTransition은 이제 대기 상태를 자동으로 처리하기 위해 비동기 함수를 지원합니다 [...]"
```

**인용 규칙:**

- 단축 URL이 아닌 전체 URL 사용
- 가능한 경우 앵커(anchor)가 포함된 딥 링크(deep link) 선호 (예: `/useActionState` 보다는 `/useActionState#usage`) — 앵커는 최상위 페이지보다 문서 구조 변경 시 더 잘 살아남습니다.
- 명확하지 않은 결정을 뒷받침할 때는 관련 구절을 인용하세요.
- 플랫폼 기능을 추천할 때는 브라우저/런타임 지원 데이터를 포함하세요.
- 패턴에 대한 문서를 찾을 수 없는 경우 명시적으로 밝히세요:

```
미검증: 이 패턴에 대한 공식 문서를 찾을 수 없습니다.
이것은 학습 데이터에 기반한 것이며 구식일 수 있습니다.
프로덕션에 사용하기 전에 검증이 필요합니다.
```

검증할 수 없는 것에 대해 정직하게 말하는 것이 거짓된 확신보다 가치 있습니다.

## 일반적인 합리화

| 합리화 | 실제 |
|---|---|
| "이 API에 대해 확신이 있어요" | 확신은 증거가 아닙니다. 학습 데이터에는 현재 버전에서는 깨지는 구식 패턴이 포함되어 있을 수 있습니다. 검증하세요. |
| "문서를 가져오는 것은 토큰을 낭비해요" | API를 환각(hallucination)하는 것이 더 큰 낭비입니다. 사용자가 한 시간 동안 디버깅한 끝에 함수 시그니처가 바뀐 것을 알게 됩니다. 한 번의 가져오기가 몇 시간의 재작업을 방지합니다. |
| "문서에 내가 필요한 게 없을 거예요" | 문서에서 다루지 않는다면 그것도 가치 있는 정보입니다 — 그 패턴이 공식적으로 권장되지 않을 수 있습니다. |
| "오래되었을 수도 있다고 언급만 할게요" | 면책 조항은 도움이 되지 않습니다. 검증하고 인용하거나, 미검증 상태임을 분명히 밝히세요. 애매하게 말하는 것이 최악의 옵션입니다. |
| "간단한 작업이라 확인할 필요 없어요" | 잘못된 패턴을 가진 간단한 작업이 템플릿이 됩니다. 사용자는 당신의 지원 중단된 폼 핸들러를 10개의 컴포넌트에 복사한 후에야 최신 방식이 존재함을 발견하게 됩니다. |

## 레드 플래그 (Red Flags)

- 해당 버전에 대한 문서를 확인하지 않고 프레임워크 특정 코드를 작성함
- 출처를 인용하는 대신 API에 대해 "믿습니다" 또는 "생각합니다"라고 말함
- 어떤 버전에 적용되는지 모른 채 패턴을 구현함
- 공식 문서 대신 Stack Overflow나 블로그 포스트를 인용함
- 학습 데이터에 나타난다는 이유로 지원 중단된 API를 사용함
- 구현 전에 `package.json` / 의존성 파일을 읽지 않음
- 프레임워크 특정 결정에 대해 출처 인용 없이 코드를 전달함
- 한 페이지만 관련이 있는데 문서 사이트 전체를 가져옴

## 검증 (Verification)

소스 주도 개발로 구현한 후 다음을 확인하세요:

- [ ] 의존성 파일에서 프레임워크 및 라이브러리 버전을 식별함
- [ ] 프레임워크 특정 패턴에 대해 공식 문서를 가져옴
- [ ] 모든 출처는 블로그 포스트나 학습 데이터가 아닌 공식 문서임
- [ ] 코드가 현재 버전의 문서에 표시된 패턴을 따름
- [ ] 사소하지 않은 결정에는 전체 URL을 포함한 출처 인용이 포함됨
- [ ] 마이그레이션 가이드를 확인하여 지원 중단된 API가 사용되지 않았음을 확인함
- [ ] 문서와 기존 코드 간의 충돌을 사용자에게 알림
- [ ] 검증할 수 없는 것은 명시적으로 미검증 상태로 표시함

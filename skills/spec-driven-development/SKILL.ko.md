# 스펙 주도 개발 (Spec-Driven Development)

## 개요 (Overview)

코드를 작성하기 전에 구조화된 스펙(사양서)을 먼저 작성하세요. 스펙은 당신과 인간 엔지니어 사이의 공유된 단일 진실 공급원(source of truth)입니다 — 우리가 무엇을 왜 구축하는지, 그리고 어떻게 완료되었음을 알 수 있는지를 정의합니다. 스펙 없는 코드는 추측일 뿐입니다.

## 사용 시점

- 새로운 프로젝트나 기능을 시작할 때
- 요구 사항이 모호하거나 불완전할 때
- 변경 사항이 여러 파일이나 모듈에 걸쳐 있을 때
- 아키텍처 결정을 내리기 직전일 때
- 작업 구현에 30분 이상 걸릴 것으로 예상될 때

**사용하지 않는 경우:** 한 줄 수정, 오타 교정, 또는 요구 사항이 명확하고 자기 완결적인 변경.

## 게이트 워크플로우 (The Gated Workflow)

스펙 주도 개발은 4단계로 구성됩니다. 현재 단계가 검증될 때까지 다음 단계로 진행하지 마세요.

```
스펙 정의 (SPECIFY) ──→ 계획 (PLAN) ──→ 작업 분해 (TASKS) ──→ 구현 (IMPLEMENT)
   │                  │                │                  │
   ▼                  ▼                ▼                  ▼
  인간 리뷰            인간 리뷰          인간 리뷰            인간 리뷰
   및 승인              및 승인            및 승인              및 승인
```

### 1단계: 스펙 정의 (Specify)

고수준의 비전에서 시작하세요. 요구 사항이 구체화될 때까지 인간에게 명확한 질문을 던지세요.

**즉시 가정을 드러내세요.** 스펙 내용을 작성하기 전에 당신이 무엇을 가정하고 있는지 나열하세요:

```
내가 하고 있는 가정들:
1. 이것은 웹 애플리케이션입니다 (네이티브 모바일이 아님)
2. 인증은 세션 기반 쿠키를 사용합니다 (JWT 아님)
3. 데이터베이스는 PostgreSQL입니다 (기존 Prisma 스키마 기준)
4. 최신 브라우저만 타겟팅합니다 (IE11 지원 안 함)
→ 지금 바로 잡아주시지 않으면 이대로 진행하겠습니다.
```

모호한 요구 사항을 혼자서 조용히 채우지 마세요. 스펙의 목적은 코드가 작성되기 *전*에 오해를 드러내는 것입니다 — 가정은 오해의 가장 위험한 형태입니다.

**다음 6가지 핵심 영역을 포함하는 스펙 문서를 작성하세요:**

1. **목표 (Objective)** — 무엇을 왜 구축하나요? 사용자는 누구인가요? 성공한 모습은 어떤 것인가요?

2. **명령어 (Commands)** — 도구 이름만이 아니라 플래그가 포함된 실행 가능한 전체 명령어.
   ```
   빌드: npm run build
   테스트: npm test -- --coverage
   린트: npm run lint --fix
   개발: npm run dev
   ```

3. **프로젝트 구조 (Project Structure)** — 소스 코드가 위치하는 곳, 테스트가 가는 곳, 문서가 속한 곳.
   ```
   src/           → 애플리케이션 소스 코드
   src/components → React 컴포넌트
   src/lib        → 공유 유틸리티
   tests/         → 단위 및 통합 테스트
   e2e/           → 엔드 투 엔드 테스트
   docs/          → 문서
   ```

4. **코드 스타일 (Code Style)** — 당신의 스타일을 보여주는 실제 코드 스니펫 하나가 그것을 설명하는 세 단락보다 낫습니다. 명명 규칙, 포맷팅 규칙 및 좋은 출력 예시를 포함하세요.

5. **테스트 전략 (Testing Strategy)** — 어떤 프레임워크를 사용할지, 테스트 위치, 커버리지 기대치, 어떤 관심사에 대해 어떤 테스트 레벨을 사용할지.

6. **경계 (Boundaries)** — 3계층 시스템:
   - **항상 할 것 (Always do):** 커밋 전 테스트 실행, 명명 규칙 준수, 입력 검증
   - **먼저 물어볼 것 (Ask first):** 데이터베이스 스키마 변경, 의존성 추가, CI 설정 변경
   - **절대 하지 말 것 (Never do):** 비밀 정보(secrets) 커밋, vendor 디렉토리 수정, 승인 없이 실패하는 테스트 제거

**스펙 템플릿:**

```markdown
# 스펙: [프로젝트/기능 이름]

## 목표
[무엇을 왜 구축하는지. 사용자 스토리 또는 수락 기준.]

## 기술 스택
[프레임워크, 언어, 버전이 포함된 주요 의존성]

## 명령어
[빌드, 테스트, 린트, 개발 — 전체 명령어]

## 프로젝트 구조
[설명이 포함된 디렉토리 레이아웃]

## 코드 스타일
[예시 스니펫 + 주요 컨벤션]

## 테스트 전략
[프레임워크, 테스트 위치, 커버리지 요구 사항, 테스트 레벨]

## 경계
- 항상 할 것: [...]
- 먼저 물어볼 것: [...]
- 절대 하지 말 것: [...]

## 성공 기준
[이것이 완료되었음을 알 수 있는 방법 — 구체적이고 테스트 가능한 조건]

## 열린 질문
[인간의 입력이 필요한 미결정 사항]
```

**지침을 성공 기준으로 재구성하세요.** 모호한 요구 사항을 받으면 구체적인 조건으로 변환하세요:

```
요구 사항: "대시보드를 더 빠르게 만드세요"

재구성된 성공 기준:
- 4G 연결에서 대시보드 LCP < 2.5초
- 초기 데이터 로드가 500ms 이내에 완료됨
- 로드 중 레이아웃 시프트 없음 (CLS < 0.1)
→ 이것들이 올바른 목표인가요?
```

이렇게 하면 "더 빠르게"가 무엇인지 추측하는 대신 명확한 목표를 향해 루프를 돌고, 재시도하고, 문제를 해결할 수 있습니다.

### 2단계: 계획 (Plan)

검증된 스펙을 바탕으로 기술적인 구현 계획을 생성하세요:

1. 주요 컴포넌트와 그 의존성 식별
2. 구현 순서 결정 (무엇을 먼저 구축해야 하는지)
3. 리스크 및 완화 전략 메모
4. 병렬로 구축할 수 있는 것과 순차적으로 구축해야 하는 것 식별
5. 단계별 검증 체크포인트 정의

계획은 리뷰 가능해야 합니다: 인간이 그것을 읽고 "네, 올바른 접근 방식입니다" 또는 "아니요, X를 변경하세요"라고 말할 수 있어야 합니다.

### 3단계: 작업 분해 (Tasks)

계획을 개별적이고 구현 가능한 작업으로 분해하세요:

- 각 작업은 하나의 집중된 세션에서 완료 가능해야 함
- 각 작업에는 명시적인 수락 기준이 있음
- 각 작업에는 검증 단계(테스트, 빌드, 수동 확인)가 포함됨
- 작업은 인지된 중요도가 아니라 의존성에 따라 순서가 정해짐
- 어떤 작업도 약 5개 이상의 파일을 변경해서는 안 됨

**작업 템플릿:**
```markdown
- [ ] 작업: [설명]
  - 수락 기준: [완료 시 충족되어야 할 조건]
  - 검증 방법: [확인 방법 — 테스트 명령어, 빌드, 수동 확인]
  - 관련 파일: [수정될 파일 목록]
```

### 4단계: 구현 (Implement)

`incremental-implementation.ko.md` 및 `test-driven-development.ko.md` 스킬을 따라 작업을 하나씩 실행하세요. `context-engineering.ko.md`를 사용하여 에이전트에게 전체 스펙을 쏟아붓는 대신 각 단계에서 적절한 스펙 섹션과 소스 파일을 로드하세요.

## 스펙 유지하기

스펙은 일회성 결과물이 아니라 살아있는 문서입니다:

- **결정이 바뀔 때 업데이트하세요** — 데이터 모델을 변경해야 함을 발견하면 먼저 스펙을 업데이트한 다음 구현하세요.
- **범위가 바뀔 때 업데이트하세요** — 추가되거나 삭제된 기능은 스펙에 반영되어야 합니다.
- **스펙을 커밋하세요** — 스펙은 코드와 함께 버전 관리 시스템에 속합니다.
- **PR에서 스펙을 참조하세요** — 각 PR이 구현하는 스펙 섹션에 링크를 거세요.

## 일반적인 합리화

| 합리화 | 실제 |
|---|---|
| "이것은 간단해서 스펙이 필요 없어요" | 간단한 작업은 *긴* 스펙이 필요하지 않지만, 여전히 수락 기준은 필요합니다. 두 줄짜리 스펙도 괜찮습니다. |
| "코딩 후에 스펙을 작성할게요" | 그것은 사양이 아니라 문서화(documentation)입니다. 스펙의 가치는 코드 작성 *전*에 명확성을 강제하는 데 있습니다. |
| "스펙 작성이 우리를 느리게 할 거예요" | 15분의 스펙 작성이 몇 시간의 재작업을 방지합니다. 15분의 폭포수 방식이 15시간의 디버깅보다 낫습니다. |
| "요구 사항은 어차피 바뀔 거예요" | 그래서 스펙은 살아있는 문서여야 합니다. 오래된 스펙이라도 아예 없는 것보다는 낫습니다. |
| "사용자는 자신이 무엇을 원하는지 알고 있어요" | 명확한 요청조차도 암묵적인 가정을 포함합니다. 스펙은 그러한 가정을 드러냅니다. |

## 레드 플래그 (Red Flags)

- 서면 요구 사항 없이 코드 작성을 시작함
- "완료"가 무엇을 의미하는지 명확히 하기 전에 "그냥 구축을 시작할까요?"라고 물음
- 스펙이나 작업 목록에 언급되지 않은 기능을 구현함
- 문서화하지 않고 아키텍처 결정을 내림
- "무엇을 구축할지 뻔하니까" 스펙 작성을 건너뜀

## 검증 (Verification)

구현을 진행하기 전에 다음을 확인하세요:

- [ ] 스펙이 6가지 핵심 영역을 모두 다루고 있음
- [ ] 인간이 스펙을 리뷰하고 승인함
- [ ] 성공 기준이 구체적이고 테스트 가능함
- [ ] 경계 (항상 할 것/먼저 물어볼 것/절대 하지 말 것)가 정의됨
- [ ] 스펙이 저장소의 파일로 저장됨
